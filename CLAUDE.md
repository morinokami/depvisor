# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

depvisor prepares Dependabot-style dependency-update PRs, shipped as a GitHub **composite action** (`action.yml`). It does what a human does to such a PR — apply the update, fix whatever it breaks, verify, and open the PR — but splits the work by trust: the dependency bump, install, and **all** verification are **deterministic**, and an LLM is used only for the two jobs that need judgment — fixing source breakage when the deterministic checks fail (the **fixer**), and writing the PR's reviewer digest (the **digest**). Keep `README.md` in sync when behavior changes — and `start.md` too: it is the agent-facing setup guide the README tells AI coding agents to fetch (by raw URL, so it must stay self-contained), and it duplicates the consumer workflow template, input names, and status names on purpose.

**Whenever you change code, check whether this file (CLAUDE.md) is still accurate — commands, architecture, invariants, gotchas — and update it in the same change if not.**

## Commands

depvisor itself uses **pnpm** (the default fixture is an npm project; a pnpm variant exists). Node >= 22.19 required (Flue; declared in package.json `engines`).

```bash
pnpm test                                    # unit tests (node --test), no API key
node --test test/grouping.test.ts            # single test file
pnpm run typecheck && pnpm run lint && pnpm run fmt:check && pnpm run knip
                                             # CI gates (plus test) — all must pass
node src/dev/scan.ts fixtures/sample-app --verify   # deterministic core E2E, no API key
pnpm run fixture:init                        # create the throwaway fixture (required before agent runs)
pnpm run fixture:init:pnpm                   # pnpm variant → fixtures/sample-app-pnpm (tests PM detection)
pnpm run fixture:init:bun                    # bun variant → fixtures/sample-app-bun (needs a local bun)
pnpm run fixture:init:workspaces             # npm-workspaces monorepo → fixtures/sample-app-workspaces
pnpm run fixture:init:workspaces:bun         # bun-workspaces monorepo → fixtures/sample-app-bun-workspaces
pnpm run fixture:init:workspaces:pnpm        # pnpm-workspaces monorepo (catalog-pinned semver) → fixtures/sample-app-pnpm-workspaces
pnpm exec flue run update                    # agent run; needs API key + DEPVISOR_LLM_MODEL in .env (DEPVISOR_OPEN_PULL_REQUESTS_LIMIT optional)
node src/open-pr.ts                          # push + PR from every emitted payload; needs GH_TOKEN + remote
cd fixtures/sample-app && git checkout -f main && npm install        # reset fixture to green baseline
```

Fixture agent run (scans and picks groups itself, same path as CI):
`DEPVISOR_TARGET_REPO="$PWD/fixtures/sample-app" pnpm exec flue run update`
(add `DEPVISOR_OPEN_PULL_REQUESTS_LIMIT=1` to cap the run at a single PR; the default ceiling is 5)

## Architecture

One split governs everything: **deterministic code does anything needing reproducibility, idempotency, or safety — the dependency bump, the install, and all verification; an LLM does only the two jobs that need judgment — fixing source breakage when the deterministic checks fail (the fixer), and writing the PR digest — each boxed between deterministic gates.**

### Modules

- `src/core/` — deterministic, LLM-free, unit-tested: pm (package-manager detection + per-PM command table + `updatePlan`) → collect (npm/pnpm outdated) → grouping (stable keys, one package per group — `major/<name>` / `dev/<name>` / `prod/<name>`, no preset bundles) → bump (apply the update plan) → verify gate → scope gates (`checkBumpScope` over the pre-commit bump diff + `checkFixScope` over the fixer's changes) → git (two-commit split + `snapshotWorktree`/`worktreeDrift`, which prove verification did not author extra files) → pr (payload/sanitize/`composeNarrative`/labels) → github (push + `gh pr create` + labels).
  - `bump.ts` (`applyUpdatePlan`) is the deterministic executor for the `UpdatePlan` `pm.updatePlan` builds: it applies every dependency bump/install/manifest edit with LLM-free code before any agent runs, so the installed version and the manifest/branch identity are fixed by code (PR-identity parity). Commands spawn without a shell (FORCE_COLOR stripped, 15-min per-command timeout, bounded output tails); the one non-command edit is pnpm's catalog move — rewritten through the `yaml` Document API so comments/formatting survive — and it is **fail-closed** (a missing entry, an unparseable file, or a non-string value is a hard failure, never a silent skip). A `CatalogEdit` names WHICH catalog to move (`catalog: string | null`, null = the default catalog: `catalog.<name>` else `catalogs.default.<name>`, a `catalog:` specifier being pnpm's sugar for `catalog:default`; a string = `catalogs.<catalog>.<name>` only, no fallback). An `UpdatePlan` may also carry `blockers?: string[]` (pnpm only — a member declared both as a `catalog:` reference and a plain version across workspaces, or an unreadable declaring manifest); `applyUpdatePlan` fails closed to `{ ok:false, step:"plan" }` before touching anything when they are present. Any failure is returned as a value (`bump-failed`), never thrown.
  - `changelog.ts` is also here: the deterministic backing for both the fixer's bounded `fetch_release_notes` tool and the release notes the workflow injects into the digest prompt. Fixed endpoints (npm registry + GitHub Releases); it never throws, returning an "unavailable" note instead.
  - `parseGithubSlug` (same file) is what the workflow uses to attach releases/compare links to the PR body — it reads the `repository` field of the packument the run already fetched, and the workflow passes the same cached-packument slug into `fetchReleaseNotes` (`opts.slug`) for the digest's notes, so no full packument is re-downloaded per member (`resolveSourceRepo`, also here, is only the lookup's fallback when no slug was supplied — i.e. the fixer's tool path). Fail-soft: unresolved → no links. `pr.ts` charset-validates every URL part at the embed boundary because the sanitizer leaves markdown links alone.
  - `version-core.ts` is the tiny leaf every version-ordering module shares: the x.y.z `Triple`, `compareTriple`, and the loose (unanchored) `parseVersionCore`. depvisor deliberately carries no semver library, and each consumer anchors its own parse variant differently on purpose — the module header documents the four flavors (loose for registry/`outdated` strings; release-age's fully-anchored `parseStable`; advisories' start-anchored OSV parse with the `"0"` sentinel; changelog's end-anchored release-tag parse).
  - `status.ts` is the deterministic status/summary boundary for Actions log UX. Its `RunStatus` is **run-level + a `groups[]` array** (one entry per group the run touched, since a run can prepare several PRs). `update.ts` writes `pr-preview/status.json` incrementally — after every group, under the red `in-progress` run status that only a graceful finish upgrades to `completed` — so a mid-loop crash leaves a status that is consistent with the payloads emitted so far AND fails the job instead of impersonating a green run; `report-status.ts` turns it into a run-level annotation plus a per-failing-group annotation and `$GITHUB_STEP_SUMMARY` Markdown. `runFailsJob` fails the job when the run-level status OR any group status is non-benign. `RUN_OUTPUT_SCHEMA`/`toRunOutput` (same file) single-source the workflow's `output` shape — a `RunStatus` projection that drops per-group `packages`/`prUrl` — so the Flue output cannot drift from the status file. `toActionOutputs` (same file) is the sibling projection behind the composite action's `outputs:` (`status`/`failed`/`prepared_count`/`pr_urls`): **machine-shaped values only** — outputs feed consumer `${{ }}` interpolation, a workflow-command/shell-injection surface — so the status must match the kebab-case vocabulary (`^[a-z-]+$`, dropped not escaped when off-vocabulary; `failed` derives from the raw status so the gate cannot launder a failure) and PR URLs a strict charset (`^https://[A-Za-z0-9./_-]+$` — a GHES port fails it, fail-closed), never free text like `summary`; `toActionOutputs(null)` (missing status file = crash before reporting) still yields `failed="true"` so `if: always()` consumers get a signal. `RUN_STATUS_FILE` lives in the tiny `status-file.ts` leaf so `pr.ts` (which clears prior output) and `status.ts` avoid an import cycle.
    - `GroupResult.usage` is an **array of `GroupUsage`** (0–2 per group) — the deterministic **LLM-usage visibility** (display-only, unit-tested). Each entry carries a `role` (`"fixer" | "digest"`) plus token counts + `costUsd` + `model`, and is a **structural projection of Flue's `PromptResultResponse.usage`/`.model`** written by `update.ts` the moment each `session.task` returns (so core stays Flue-free — the mapping is a plain object literal in the workflow, no Flue-type import). A group runs 0–2 agent operations — the failure-path `fixer` and the per-PR `digest` — and each pushes its entry as it returns; outcomes that ran no agent (skips/hold-backs/`branch-collision`/`release-age-unavailable`/`bump-failed`) record none, and an operation that threw before returning a response (the `ResultUnavailableError` half of `no-structured-result`) records nothing for that operation. Captured **before the defensive re-parse and the verdict branch**, so a fixer `defer` (still burned tokens) reports its usage, as does a returned response whose re-parse rejected. `sumGroupUsage` totals every group AND every operation within it (null when none ran anywhere — callers render nothing rather than a misleading zero row); `renderStepSummary` shows a run-total row in the header plus a per-group row with a per-role breakdown (`fixer … + digest …`), `runLogLine` a compact `tokens=/cost=`. `costUsd` is Flue's provider-priced **estimate** (BYOK-approximate, rendered `est. ~$…`), not an invoice. **Record-only, exactly like `testChanges`**: `parseGroup` preserves it across the open-pr read→rewrite round-trip and it is deliberately **NOT** in `RUN_OUTPUT_SCHEMA` (numbers only, so no token-separation/gate impact — it is written inside the token-free agent step).
  - `budget.ts` is the deterministic `open_pull_requests_limit` policy (LLM-free, unit-tested): `parseOpenPullRequestsLimit` (empty → `5`, Dependabot's open-pull-requests-limit default), `countOpenDepvisorPrs`, and `classifyGroup` (skip-up-to-date / refresh / open-new / held-back). `open_pull_requests_limit` is a **ceiling on open depvisor PRs** (Dependabot's open-pull-requests-limit model), not a per-run throughput cap — a run opens new PRs only up to the ceiling but always refreshes existing ones, and only a newly opened PR consumes a slot. Since every group is a single package, the ceiling reads as "how many independent package updates may be in flight".
  - `release-age.ts` is the deterministic `minimum_release_age` cooldown (LLM-free, unit-tested): `parseMinimumReleaseAge` (empty → `1`, `0` = explicit disable), `parseMinimumReleaseAgeExclude` (newline-separated bare names, blank/`#`-comment lines skipped, fail-closed `bad-minimum-release-age-exclude` — validated even when the cooldown is disabled so a typo fails now, not on re-enable), `fetchPackument` (fixed npm-registry endpoint, injectable fetch + timeout), and `applyReleaseAge`, which — between collect and grouping — rounds each candidate's `latest` down to the newest **stable** version that is both newer than `current` and at least N days old (publish times from packument `time` ∩ `versions`, so unpublished versions and `created`/`modified` never count; a prerelease `latest` passes only via its exact-string maturity). It recomputes `updateType` after a clamp because grouping (branch/PR identity) depends on it. **Fail-closed, unlike changelog.ts**: 404 and network error alike drop the candidate (`release-age-unavailable`, red) — this module is a defense, not display. The per-package escape hatch is `minimum_release_age_exclude` (pnpm's `minimumReleaseAgeExclude`): names in `opts.exclude` skip the fetch and the age check entirely and pass through verbatim (recorded in the result's `excluded`, green — a human opt-out for private-registry packages npmjs cannot vouch for, not a fail-open path; `heldBack` is the not-yet-mature drop). Their packument is still fetched fail-soft at PR time for links/license. The packument cache it fills is reused for the PR body's source-repo links AND for the digest's release-notes source resolution (`fetchReleaseNotes`'s `opts.slug` — no second registry hit per package).
  - `ignore.ts` is the deterministic `ignore` policy (LLM-free, unit-tested): `parseIgnore` (a `{ ok; rules } | { ok:false; invalid }` union — blank lines and full-line `#` comments skipped, every other line must be `name` or `name@<major>`, else fail-closed with the offending entries) and `applyIgnore` (a pure prepend filter splitting candidates into kept/ignored, plus `describeIgnore` for the log line). Syntax is deliberately semver-library-free: `name` drops the package always, `name@<major>` only when the candidate's `latest` major matches. It runs **first — before the cooldown clamp and grouping** (see the flow below), so an ignored package costs no packument fetch / agent run / spurious `release-age-unavailable`; the trade-off is that `name@<major>` matches the raw registry `latest`, not a clamped version (conservative — documented in the module header and README). Trust model = `verify_commands`: workflow config only, never the agent-writable target tree.
  - `suggest-features.ts` is the deterministic parse for the opt-in `suggest_features` flag (LLM-free, unit-tested): `parseSuggestFeatures` (trim, then empty/`"false"` → off, `"true"` → on, else `null` → the run-level stop `bad-suggest-features`; the exact-string, fail-closed shape of the other config knobs). When on, `update.ts` appends a paragraph to the **digest** prompt **only for a group with a minor/major member** (patch releases carry no new capability; judged on the post-clamp `updateType` grouping already used) asking the digest to surface newly added capabilities relevant to the codebase — based **only on the release notes already injected into its prompt** for the update (the cost/exposure lever: never a fetch just to suggest) — in the optional `relevant_new_features` result field (`{package, summary, codebase_relevance}[]`, added to `update.ts`'s Valibot `DigestResult`; `digest.md` is untouched). `update.ts` passes the field toward the payload **only under the same flag-on + non-patch-member condition that emitted the instruction** (`wantsSuggestions` — the schema is model-visible even without the instruction, so an off run or a patch-only group could fill it unbidden and is ignored), and `pr.ts:newFeaturesSection` renders it display-only: filtered to group members (like `notable_changes`, package label/version taken from the validated candidate), free text through the same `sanitizeSummary` treatment, capped at `MAX_NEW_FEATURE_BULLETS` (5) with an explicit omitted-count line, placed after the narrative sections and before `## Verification`. A **plain markdown section, not `<details>`**, because `sanitizePrBody` escapes every `<` at the exit boundary. **Never a gate, always green** (the only new status is the config-parse `bad-suggest-features`); adopting a feature is NOT deterministically prevented (product code is outside the scope gate, like tests), so the prompt's "report only, never modify code to adopt" is the sole guard, backstopped only by human review of the fix commit.
  - `advisories.ts` is the deterministic **security prioritization** (LLM-free, unit-tested): `fetchAdvisories` queries OSV.dev (two fixed endpoints — a `/v1/querybatch` triage of every candidate at EACH declared `current` version, then `/v1/query` per flagged (name, current) pair for full `affected` ranges), `resolvesAdvisory` decides whether an update RESOLVES an advisory (current affected AND target not — SEMVER ranges matched by x.y.z core comparison, no semver library), `prioritizeGroups` stable-promotes resolving groups to the front, and `describeAdvisories` is the log line. It probes **every** workspace-current (`Candidate.currents`), not just the merged lowest `current`, so a vulnerability carried only by a higher-versioned workspace is not hidden behind the lowest — for **npm/bun**, whose collectors report every occurrence. **pnpm is the exception**: `pnpm outdated` reports only the highest installed version and omits lower-versioned workspaces, so `currents` there is just that highest, and an advisory affecting solely an omitted lower version is missed (fail-soft: the update still fixes every workspace; only the promotion/Security hint is lost — reconstructing pnpm's per-workspace versions via `pnpm list -r` is a deliberate non-goal). **Ordering only** — it never changes which version installs, only the processing order (so branch/PR identity is untouched), and it feeds the PR body's Security column. Promotion and display are both keyed on the advisory's **GHSA id** (from OSV `id` or `aliases`, validated by `GHSA_RE`, which `pr.ts` re-validates at the embed boundary), so the unit that promotes a group is exactly the id the PR can link; a GHSA-less advisory is not counted. **Fail-soft, the deliberate opposite of `release-age.ts`**: it is an optimization, not a defense, so any OSV failure degrades to the neutral order (`ok:false`, logged, and `ADVISORIES_UNAVAILABLE_NOTE` is appended to the completed run's summary — the run stays green, so that note is the only user-visible trace of the degradation) — including a full-query failure for a package the triage already flagged, since ordering on a partial snapshot would silently mis-rank the `open_pull_requests_limit` slots — and any unevaluable range fails toward "still affected" (no promotion on a maybe). Runs against the **post-clamp** `latest`, so a fix still inside the cooldown window does not count (cooldown wins over urgency).
  - `test-changes.ts` is the deterministic **test-change visibility** (LLM-free, unit-tested), the display-only counterpart to `scope.ts`: `isTestPath` classifies a repo path against a **static** world-convention set (`test/`/`tests/`/`__tests__/`/`__mocks__/`/`spec/`/`e2e/`/`cypress/`/`playwright/` dirs, `*.test.*`/`*.spec.*`/`*_test.*` names — deliberately NOT the target's agent-writable test-runner config), `classifyTestChanges` filters a `diffNumstat` result to the test subset, and `formatNumstatLines` renders the delta (`binary` for null counts). The scope gate **cannot deny tests** (adapting a test to a changed API is a legitimate update, indistinguishable from a poisoned fixer weakening an assertion), so instead the workflow classifies the committed `base..HEAD` diff and, when tests changed, adds a ⚠️ warning section to the PR body and step summary — visibility, never a gate (**no new failing status**, always green). Completeness is a non-goal; the wording never implies an empty section proves no test was touched. `git.ts:diffNumstat` is the deterministic backing (parses `git diff --numstat --no-renames -z`; binary `-` → null counts). It uses **`--no-renames` on purpose**: git's default rename detection would collapse a moved test (`test/a.test.ts` → `src/a.ts`) into one record keyed on the non-test destination, hiding that a test was dropped out of the test globs (silently disabled while verification still passes); `--no-renames` decomposes the move into a delete of the old test path plus an add, so the test path always surfaces. Paths are charset-validated (`pr.ts:isDisplayablePath`) before embedding — filenames can hold backticks/spaces (observed in real `git diff`) that would break out of a markdown code span — with any dropped path still counted.
  - `license.ts` is the deterministic **license-change visibility** (LLM-free, unit-tested), a display-only sibling of `test-changes.ts`: `versionLicense` reads one published version's `license` field from a `release-age.ts` `Packument` as a **plain string** (the deprecated `{ type }` object form, the ancient `licenses` array, a missing/blank field, and an absent version all read as `null` = unknown), `classifyLicenseChanges` emits one `LicenseChange` per DISTINCT `from → to` license pair a member crosses — it checks the target against EVERY declared `Candidate.currents` (mirroring `advisories.ts`), not just the merged lowest `current`, so a relicense carried only by a higher-versioned workspace is not hidden; a member can therefore yield several rows, with repeated `from → to` deduped (the pnpm per-workspace-collapse caveat is the same as advisory matching). Only a KNOWN string license on both sides counts (exact string compare, no SPDX parsing — the "never interpret" policy; an unknown side is skipped so an object-form "MIT" vs a string "MIT" never manufactures a phantom change), and `describeLicenseChanges` is the log line — which control-sanitizes the registry license strings (`\p{Cc}` runs collapsed to spaces, length-capped) because the CI log is a raw stream where an embedded newline could forge an Actions `::command` (the log-boundary counterpart of `LICENSE_RE`). The workflow renders a ⚠️ warning section in the PR body (`pr.ts:licenseChangesSection`); the license strings are registry-derived = untrusted, so they are charset-gated by `pr.ts`'s `LICENSE_RE` (SPDX-shaped `[A-Za-z0-9 .+()-]`, so a backtick/`[`/`|` cannot break out of the code span or table row) with any dropped change still counted. **Fail-open, the deliberate opposite of `release-age.ts`**: display, not a defense, so an unresolvable license shows nothing rather than blocking a PR. It reuses the packument the cooldown/source-repo links already fetched (**no extra registry hit**; when the cooldown is disabled `update.ts` fetches each member's full packument once — the same round-trip the old `resolveSourceRepo` made — and derives both the source slug and the license from it). PR-body-only today (unlike `test-changes.ts`, it is NOT surfaced in the step summary — hence `isDisplayableLicense` is unexported, mirror the test-change path if that changes).
  - `install.ts` (`runInstall`) is the shared install spawn used by both `install-target.ts` (pre-agent) and the group-boundary reset in `update.ts`. It carries a 15-minute timeout (mirroring verify's per-step timeout) so a hung install cannot eat the CI job.
  - `pr.ts:deriveLabels` is the deterministic **PR-label policy** (LLM-free, unit-tested): `depvisor` always, the update's `semver:<patch|minor|major>` (groups are singletons today so this is simply that package's level; the top-of-members fold is the plural handling a future user-declared `groups` config needs — majors stay isolated per package, so even then a group mixes at most minor+patch), `security` when a member resolves an advisory (same `advisories.resolvedByPackage` map that feeds the Security column), `dev-dependencies` when every member is a dev dep. The set rides in `PrPayload.labels` (written by the tokenless step) and is applied **only** in the token-holding step (`github.ts:applyLabels`), which first re-validates it against a fixed allowlist (`pr.ts:sanitizeLabels`, the label counterpart of `sanitizePrBody` — drops anything off-vocabulary, incl. flag-shaped/injection strings, dedupes, stabilizes). Application is **fail-soft and decoupled from PR creation**: it runs AFTER `gh pr create` (labels are NEVER passed to create — `gh pr create --label <unknown>` aborts creation, turning a missing label into a lost PR), ensuring each label with `gh label create` (no `--force`, so a user's same-named label keeps its color/description) then adding it via `gh pr edit --add-label` **one label per edit**, on both the created and refreshed paths. Per-label (not a single batched `--add-label`) is deliberate: `gh pr edit` resolves all requested labels up front and errors on any unknown one, so a batch is all-or-nothing and one label a transient create failure left missing would drop even the labels that exist. Both creating and applying a label need only `pull-requests: write` — GitHub's label endpoint accepts `issues` OR `pull-requests` write (verified empirically against the standard `GITHUB_TOKEN`), so **no `issues: write` is required**; each label is applied independently and an unappliable one is logged and skipped, never fatal. Labels never touch branch/group keys, membership, or the installed version, so PR identity is untouched (like `advisories.ts`, ordering/display only).
- `src/agents/depvisor.ts` (+ the `fixer.md` / `digest.md` instructions) and `src/workflows/update.ts` — the only LLM-driven code (`src/tools/` and `flue.config.ts` also touch Flue APIs, but stay LLM-free).
  - The `defineAgent` root, `depvisor`, lives in `src/agents/` per Flue's discovery convention (filename = agent name); it exports no `route`, so it is never HTTP-exposed. It is **never prompted directly** — it exists only to select Flue's default in-memory virtual sandbox (`cwd: /workspace`) and hold the two subagent profiles the workflow delegates to. The virtual sandbox's built-in fs/shell cannot see the host; every host-repo capability is a profile-owned custom tool.
  - The two roles are `defineAgentProfile`s (`subagents: [fixer, digest]`), each run via `session.task(prompt, { agent: "fixer" | "digest", result })`:
    - The **fixer** (instructions from `fixer.md`) gets the repo-jailed read tools plus `write_repo_file` / `replace_repo_text` / `remove_repo_file`, and the bounded `fetch_release_notes` (`src/tools/release-notes.ts`, backed by `core/changelog.ts`) — the single narrow door for untrusted external text (never raw agent HTTP). It runs ONLY on the failure path (post-bump verification red): the bump is already applied and committed, so it edits source/tests and may never touch manifests/lockfiles/config — the scope gate (`checkFixScope`) enforces that. It has **no host shell and cannot run verification itself**; the workflow owns the sole authoritative post-fix run.
    - The **digest** gets only `list_repo_files` / `read_repo_file` / `search_repo` (`src/tools/repo-files.ts`) and no host write/exec capability. Release notes are fetched deterministically (`core/changelog.ts`) and injected into its prompt (capped per package). It runs for EVERY prepared PR strictly AFTER commits are sealed, reading the codebase through those bounded tools to ground the reviewer narrative. This is capability enforcement, not a prompt-only "stay read-only" promise.
  - `src/tools/repo-files.ts` is the host bridge for both roles: every path must be repo-relative, resolves below the real target root even through symlinks, and `.git` is rejected; reads/search/list outputs and writes are bounded. Keep digest's profile on `repoReadTools` only. Both `.md` files import `with { type: "markdown" }` (Flue's convention); tool modules are not auto-discovered, so `depvisor.ts` attaches them explicitly.
- `src/shared/target.ts` — the target checkout value shared by the deterministic workflow and the bounded repo tools: `REPO` (CI checkout / throwaway fixture locally). It is deliberately NOT the virtual agent sandbox cwd.
- `src/open-pr.ts` — the deterministic, token-holding entrypoint: it enumerates **every** payload under `pr-preview/payloads/` (filename = processing order) and pushes + `gh pr create`s each via `openPrWithGh`, recording each payload's outcome in the status file via `status.ts:recordGroupOutcome` — the PR URL, a blocked-takeover note (`open-pr-blocked`, stays green), or an open-pr failure (`open-pr-failed`). `github.ts` reserves the `blocked` result for the ONE expected policy stop (a human's commits at the remote branch tip); every other push-boundary refusal — non-depvisor branch or base, a foreign committer in the local range, non-network remote — is a `failed` result (tampering/misconfiguration), because riding the green blocked path would end the job green with no PR opened, the silent no-PR outcome the status design promises to surface. `recordGroupOutcome` patches the group whose branch matches and otherwise **appends a synthetic entry** (an unreadable payload — branch unknown — or a payload naming a branch the run never recorded): without the append, the stale `pr-prepared` entry would keep the report step's `failed` output/annotations green while this step's non-zero exit turns the job red. Payload read-back is JSON.parse + `pr.ts:parsePrPayload` **shape validation** (a JSON-parseable non-payload like `{}` counts as unreadable — it would otherwise throw at `payload.branch.startsWith` mid-push), and the `openPrWithGh` call is additionally wrapped per payload so an unexpected throw (fs/temp-dir errors; openPrWithGh reports expected failures as values but is not exception-free) records that group's `open-pr-failed` instead of killing the loop. One payload's failure does not stop the rest; it exits 1 if any failed. The ONLY command that needs `GH_TOKEN`; in CI it runs as its own Action step. `openPrWithGh` is per-payload self-contained (fresh clone + authorization checks + deterministic label application), so multi-PR is just calling it N times — do not add cross-payload state to it.
- `src/report-status.ts` — the deterministic Actions log UX entrypoint. It reads `pr-preview/status.json`, emits a run-level annotation (level from `runFailsJob`) plus one error annotation per failing group, appends a per-group Markdown step summary, and exits non-zero via `runFailsJob`. It also writes `toActionOutputs` to `$GITHUB_OUTPUT` (uniform heredoc, `crypto.randomUUID` delimiter — the standard delimiter-collision defense), **on both the normal and the missing/corrupt-status-file path, before any `exit(1)`** — a failed step's `GITHUB_OUTPUT` writes still reach the composite `outputs:` mapping (`id: report` in action.yml), which is what makes outputs usable from `if: always()` consumer steps; the one platform caveat is that nesting the action inside another composite loses step outputs (actions/runner#2009, documented in the README). `readRunStatus` fails toward null on a corrupt/truncated/non-object status file (it is untrusted at read-back) so the reporter and open-pr's per-payload loop degrade instead of crashing; missing and corrupt differ only in the reporter's message.
- `src/check-credentials.ts` — the plain-node entrypoint for the credentials gate: fails the job with `::error` when `core/credentials.ts` finds persisted credentials in the target checkout. Runs as the first target-touching Action step — **before** the target install (whose lifecycle scripts must not see a token either) and regardless of `install_command`.
- `src/install-target.ts` — the plain-node entrypoint behind `install_command: auto`: detects the target's PM and runs the matching lockfile-faithful install (`npm ci` / `pnpm install --frozen-lockfile`) before the agent step.
  - It fails the job with a clear `::error` for unsupported/ambiguous PMs AND for repos without a committed lockfile — a bare `npm install` would create the lockfile and dirty the pre-agent tree, which preflight would later reject as an illegible `dirty-tree`.
- `src/dev/scan.ts` — a developer-only tool (no key) that runs detect → collect → group → verify against a repo and prints it. NOT part of the CI/composite-action flow, hence its home under `dev/`. `--minimum-release-age=<days>` opt-ins the cooldown clamp (hits the real npm registry; the default scan stays offline).

### The update flow

The workflow (`flue run update`) imports the `depvisor` root agent and drives the deterministic bump/install/verify itself, delegating only the fixer and digest to its subagent profiles. Preflight, PM detection, and collect+group run **once**; then it **loops over groups** up to the `open_pull_requests_limit` open-PR ceiling (`budget.ts`):

0. Preflight (repo root, credentials, clean tree, PM detection, base, verify steps), `parseOpenPullRequestsLimit`, `parseMinimumReleaseAge`, `parseMinimumReleaseAgeExclude`, `parseIgnore`, and `parseSuggestFeatures` — any failure is a run-level stop, no loop.
1. Collect, then the `ignore` filter (`ignore.ts`, when rules are set), then the `minimum_release_age` clamp (`release-age.ts`, when N > 0), then grouping. Ignore runs **first** so an ignored package never costs a packument fetch, an agent run, or a red `release-age-unavailable` — its `name@<major>` therefore matches the raw registry `latest` (conservative; documented). The clamp must sit between ignore and grouping because it can reclassify `updateType`, which the group key (branch/PR identity) depends on. Candidates the registry cannot vouch for are dropped here and later recorded as `release-age-unavailable` group entries (branch/group null, red); ignore drops, cooldown clamps, and not-yet-mature exclusions are all normal operation (green), reported via the run summary/log only. Then **security prioritization** (`advisories.ts`, once, after grouping): `fetchAdvisories` + `prioritizeGroups` stable-promote groups whose update resolves an OSV advisory to the front so the budget below spends slots on security fixes first — ordering only (identity untouched), fail-soft (an OSV outage keeps the neutral order), evaluated on the post-clamp `latest`. The resolved-advisory map also drives the PR body's Security column.
2. Count open depvisor PRs from the snapshot (`gh pr list --limit 1000`; the ceiling **fails open** if the snapshot is truncated or absent — locally, or on repos with more open PRs than the limit) → `newSlots = max(0, open_pull_requests_limit - openCount)`. For each group in stable order, first a **branch-collision guard** (distinct group keys can slugify to the same branch, e.g. `prod/@babel/core` vs `prod/babel-core`; colliders after the first are recorded `branch-collision`, red, and skipped so they cannot `ensureBranch`-reset the first group's commits away), then `classifyGroup` decides: skip-up-to-date / refresh / open-new / held-back (`held-back-by-limit`, green).
3. Between processed groups, reset the tree to base (`resetToBase` = `git checkout -f base` + `clean -fd`) and reinstall (`runInstall` with the resolved reset command) so node_modules returns to the base lockfile state. This reinstall also runs under `install_command: skip` (skip only skips the pre-agent install step); with `skip` **and** no committed lockfile no reset command exists, so every processable group after the first is recorded `reinstall-unavailable` (red — a fixable config gap, not the ceiling). Every ref is snapshotted (`git.ts:snapshotRefs`) **before** the reinstall — the iteration's first untrusted execution (the base lockfile's lifecycle scripts run with `.git` reachable) — and checked right after it, success or failure; drift is `unexpected-commits` (restored, group failed, run continues). A successful reinstall must also leave the tracked/untracked tree clean; any file side effect is `reset-failed`. See the ref-integrity invariant below.
4. Baseline verification **per processed group** — the ref check runs right after it (its scripts execute the base tree's code), before the red/green verdict is honored, and the tree must still be clean (verification is a gate, never a source author); then `baseline-red` on the first processed group (base itself is broken/dirty), `reset-failed` on a later one (the reset was incomplete). Both are fail-closed, run-level stops.
5. **Deterministic bump** (`bump.ts`'s `applyUpdatePlan` on `pm.updatePlan(members, REPO, { pinExact })`): capture the IMMUTABLE pre-bump sha (the branch tip trusted code just created — also the maintained snapshot's entry for this branch), then apply the update, install, and manifest edits (incl. pnpm's catalog YAML edit) with LLM-free code. The bump ran the newly installed version's lifecycle scripts with `.git` reachable — exactly the supply-chain vector the deny list exists for — so refs AND HEAD are checked against the snapshot right after it, **success or failure** (a failing install still ran its scripts): a script `git commit`-ing its edits (so the working-tree gates never see them) or moving any other ref is `unexpected-commits`, restored from the snapshot. Only then a failure is `bump-failed` (red, per-group, the run continues — the fixer cannot help, it may not touch manifests). No tree changes → `no-changes`.
6. **Bump-scope gate, then bump commit, then the fast/fixer split.** Before committing, `checkBumpScope(REPO, preBumpSha, members, plan.catalogEdits)` (`core/scope.ts`) diffs the working tree against the pre-bump sha (never a movable ref name — the same scripts it defends against can move refs) and allows only what the plan itself writes to the files that WOULD enter the bump commit — member values moving to exactly `latest`/`^latest`/`~latest` (a whole-string grammar, so `npm:evil@<latest>` alias redirects and `git:`/`file:`/URL specifiers all fail), pnpm-workspace.yaml changes keyed to the plan's own `catalogEdits` (a same-named entry in a catalog the plan did not target may not change), and (uninspected) lockfiles; anything else (an install lifecycle script that rewrote `scripts`/`overrides`/`trustedDependencies`/a non-catalog workspace key, a de-catalog, a planted package.json) is `scope-violation` (red, nothing committed) — this closes the window where such tampering would ride along in the "mechanical" bump commit, invisible to `checkFixScope` (which diffs FROM the bump commit). Then commit the mechanical bump (`deps: bump <pkgs>`) _before any agent runs_ — the AI provably wrote none of it — via `commitPaths(manifestBumpPaths(…))`; if that returns null (the tree changed but no manifest/lockfile did, e.g. a lifecycle side effect) it is `bump-failed` (red, the changed files discarded). **Immediately after that commit the tree must be clean**: any source/test leftover from install scripts is `scope-violation`, never a second commit. Advance the maintained ref snapshot's branch entry to the bump sha, run full verification, re-check refs, and again require a clean tree (verification is a gate, not a source author):
   - **Fast path** (verification green): no fixer runs and no second commit exists.
   - **Fixer path** (verification red): `session.task(..., { agent: "fixer", result: FixerResult })` gets a bounded prompt — targets, **manifest diff hunks only** (`git.ts:manifestDiff`; lockfiles never as hunks), failing steps with bounded output tails (`verify.ts` captures a `tail` internally; `stripVerifyTails` removes it before persistence), the source-only constraint, and verify command names. The fixer inspects/edits only through the repo-jailed custom tools and cannot run checks itself. Verdict is `fixed`/`defer` (a `defer` discards leftovers → `deferred`; no validated result → fail-closed `no-structured-result`).
7. **Fixer-path gates** — authoritative regardless of what the fixer claims: `checkFixScope(REPO, bumpSha)` runs before verification; `snapshotWorktree` captures the exact allowed fixer diff; full re-verification runs; refs/HEAD are checked; `worktreeDrift` rejects ANY verification-authored content/mode/path change; then `checkFixScope` runs a second time **immediately before `commitAll`** so no denied path can cross the commit boundary. A red check is `scope-violation`/`verification-failed`; only then is the fix commit created as the second commit.
8. Classify the committed `base..HEAD` diff (`test-changes.ts`, display only — a ⚠️ PR-body/step-summary warning when a test file changed, no gate) → classify per-package license changes from fetched packuments (`license.ts`) → run the **digest** (`session.task(..., { agent: "digest", result: DigestResult })`, always, strictly after commits are sealed; fail-soft on `FlueError` / `ResultUnavailableError` / Valibot rejection). The digest has only repo-jailed read tools inside a virtual sandbox, so release-note prompt injection cannot write the target, depvisor's action checkout, payload directory, or later token-holding entrypoint. The post-digest ref/tree seal remains defense in depth for delayed target processes; unexpected drift restores the sealed state and drops the report. Then compose the narrative and emit the payload/status.

- Per-group outcomes (`defer` → `deferred`, `no-structured-result`, `unexpected-commits`, `scope-violation`, `verification-failed`, `bump-failed`, `no-changes`, `branch-collision`, `reinstall-unavailable`, `release-age-unavailable`) record that group and **continue to the next group**; leftover changes from a defer or a failed bump are discarded deterministically by the next group's reset, not trusted away. Only preflight, `bad-open-pull-requests-limit`, `bad-minimum-release-age`, `bad-minimum-release-age-exclude`, `bad-ignore`, `bad-suggest-features`, `baseline-red`, and `reset-failed` stop the whole run.
- The composite action reports the status file after the token-holding open-pr step. Run-level `completed`/`no-updates` and group-level `pr-prepared`, `pr-up-to-date`, `deferred`, `open-pr-blocked` (a human took over the PR branch), and `held-back-by-limit` stay green; `baseline-red`, `reset-failed`, `bad-open-pull-requests-limit`, `bad-minimum-release-age`, `bad-minimum-release-age-exclude`, `bad-ignore`, `bad-suggest-features`, `no-verify-scripts`, `bump-failed`, `scope-violation`, `verification-failed`, `missing-base-branch`, `reinstall-unavailable`, `release-age-unavailable`, `branch-collision`, `open-pr-failed`, and the `in-progress` crash marker fail the job — a `completed` run is still red if any group failed — so silent no-PR outcomes notify users.
- `open_pull_requests_limit` is a ceiling on **open depvisor PRs** (Dependabot's open-pull-requests-limit model, default `5` like Dependabot's): it never opens a new PR while the ceiling is already met by pre-existing open PRs. Refreshing an existing PR (its targets drifted) is unlimited and consumes no slot.

### Invariants

Preserve these when changing the flow above.

- **Token separation is structural, not instruction-based**: the agent step holds only the LLM key; GitHub tokens exist only in the snapshot and open-pr steps, with `persist-credentials: false` on checkout. Never hand the agent step a token or let the agent run git/GitHub operations.
  - The deterministic bump (`bump.ts`) and the multi-PR loop's group-boundary reinstall (`runInstall`) both run the target's lifecycle scripts inside the token-free agent step. This adds **no new exposure class**: install scripts already run there (the pre-agent install and the between-groups reinstall), and no GitHub token is ever present in that step.
  - **Agent host isolation is capability-based**: the root uses Flue's in-memory virtual sandbox, so built-in fs/shell cannot reach the runner. `repo-files.ts` is the only host bridge; digest receives read/list/search only, fixer additionally receives bounded repo-relative write/replace/remove, and `.git`/symlink/`..` escapes are rejected. Neither role can rewrite depvisor's own checkout or the source the later GH_TOKEN step executes. Do not reintroduce `local()` or a host shell tool without restoring an OS-level isolation boundary first.
  - The user's checkout can defeat the separation from outside (actions/checkout defaults to `persist-credentials: true`), so `core/credentials.ts` fail-closes on persisted credentials in the checkout's repo-**local** git config: Authorization extraheaders (URL-scoped or not), userinfo in http(s) remote url/pushurl and in `url.*.insteadOf`/`pushInsteadOf` rewrites, `core.sshCommand`, repo-local credential helpers. The config read follows include directives (`--includes` in `localConfigEntries`) because checkout v6+ persists the token in a separate RUNNER_TEMP file referenced via `include.path` from the repo-local config. Enforced twice — the `check-credentials.ts` Action step, and the `persisted-credentials` preflight status covering local/non-action runs. Best-effort over known vectors, not a guarantee. Findings never include config values, and redact URL subsections from keys — both positions can carry the secret (`url.<https://token@…>.insteadOf` puts it in the key).
- **Commit identity is split on purpose** (`git.ts:commitStaged`): the **author** is the resolvable `41898282+github-actions[bot]@users.noreply.github.com` — display-only, because Vercel-class Git integrations refuse to build a PR whose commit author resolves to no GitHub account (#46) — while the **committer** stays the unclaimable `depvisor[bot]@users.noreply.github.com` sentinel (`AGENT_EMAIL`), and both push-boundary guards in `github.ts` (foreign commits in `base..branch` → `failed`, remote-tip human takeover → `blocked`) key on the committer (`%ce`), never the author. Keeping the committer unchanged is also the migration guarantee: tips of PR branches created before the split carry the sentinel in both fields, so they keep refreshing. A human rebase/amend/web-UI edit always rewrites the committer — exactly the takeover signal — so never move a guard back onto the author (trivially forged via `--author`; these guards were never cryptographic — the sha-based gates are the hard ones) and never make the committer configurable. A configurable _author_ input would be safe now, but is deliberately deferred until someone asks.
- **The mechanical bump commit is made before any LLM runs** (`update.ts`): `deps: bump <pkgs>` is committed by deterministic code the moment the bump applies, so a reviewer is structurally guaranteed the AI wrote none of it; the `fix: adapt code to <pkgs>` commit exists only when the fixer adapted source. Install/verification leftovers never fold into it: clean-tree/worktree-drift gates reject them. Never let an agent create either commit.
- **Branch name = stable group key = PR identity**: changing grouping or naming logic changes PR identity and breaks idempotency/skip-if-up-to-date.
  - The `minimum_release_age` clamp is a **known, documented exception**: a clamped candidate's `updateType` is recomputed, so a cooldown-window major can surface first as e.g. `prod/<name>` and later, once matured, move to `major/<name>` — a different branch and a new PR. The stranded earlier PR is not auto-closed and keeps occupying a `open_pull_requests_limit` slot until a human closes/merges it (README documents this; a superseded-PR closer is a recorded future feature, not present).
  - **Security prioritization (`advisories.ts`) is ordering ONLY, so it is NOT an identity exception**: it reorders groups but never touches keys, membership, or the installed version. Keep it that way — a "promote by editing the group key" shortcut would break idempotency. Its one determinism caveat is like the clamp's: OSV data is time-varying, so "same input → same order" holds only within an advisory snapshot; the sort itself is stable (`Array.prototype.sort`, Node 22) and deterministic given that snapshot.
- **Package-manager detection is pinned at preflight** (`core/pm.ts`, called once in `update.ts` preflight against the trusted base tree; also by `install-target.ts` and `dev/scan.ts`).
  - Supported: npm + pnpm + bun; yarn and ambiguous lockfile mixes fail closed (`unsupported-package-manager` / `ambiguous-package-manager`, no agent run). bun repos need the bun binary on the runner (`oven-sh/setup-bun`, version pinned — see README).
  - Never re-detect after the agent has run — lockfiles are agent-writable, so post-agent detection would let the agent switch which commands the trusted steps execute.
  - The detected `pm` drives collect, verify `run` prefixes, the deterministic bump (`pm.updatePlan`, applied by `bump.ts`), the group-boundary reinstall in the multi-PR loop (via `pm.installCommand`, when `install_command` is `auto`/`skip`), and (via `pm.lockfiles`) the lockfile half of the two-commit split's bump commit — `git.ts`'s `manifestBumpPaths` pairs those lockfiles with every changed `package.json` (root or workspace), matched by basename.
  - `pm.updatePlan(candidates, repo, { pinExact })` builds the deterministic `UpdatePlan` (`{ catalogEdits, commands, pinExact }`) that `bump.ts` executes; `pinExact: true` whenever the `minimum_release_age` cooldown is active. bun's plan changes under it (`bun add <name>@<latest>`, no `^`), because bun both writes the specifier verbatim and resolves ranges at install time — a caret range would let the install reach back into the cooldown window that the clamp just excluded. npm installs exact from `npm install <name>@<ver>` and pnpm from `pnpm -r update <name>@<ver>`, so their commands ignore the flag — but pnpm's **catalog entries** tighten under it (written exact, same reasoning as bun: a hand-written range is resolved fresh by the follow-up install).
  - **pnpm catalogs are moved by a deterministic YAML edit, not a pnpm command**: pnpm has no command that moves a catalog entry to a SPECIFIC version — `pnpm -r update <name>@<ver>` **de-catalogs** instead (rewrites each workspace's `catalog:` specifier to the plain version, leaving the catalog entry stale), and `--latest` (which does edit the catalog) rejects version specs (`ERR_PNPM_LATEST_WITH_SPEC`); both verified on pnpm 11.9. So `pnpmUpdatePlan` routes catalog-pinned members to `catalogEdits` (excluded from the update command), and `bump.ts` rewrites the `catalog`/`catalogs` entry through the `yaml` Document API, then runs `pnpm install --no-frozen-lockfile` (CI=true flips pnpm's install default to frozen, which would fail on the edited catalog). The edit is LLM-free code (its correctness is a `bump.ts` unit-test concern, not an agent-trust gate); `pm.extraBumpFiles` routes the changed pnpm-workspace.yaml into the mechanical bump commit (`git.ts:manifestBumpPaths` matches it by exact root path, not basename).
    - **Which members are catalog-pinned is decided by the actual DECLARATIONS across EVERY workspace, not a name-global scan of the catalog keys and not `Candidate.locations`**: `pnpmUpdatePlan` enumerates the workspace set itself (`pnpmWorkspaceDirs` expands pnpm-workspace.yaml's `packages` patterns — literals, `*`, `**`, `!negations`, with leading `./` normalized to pnpm's equivalent repo-relative spelling; anything fancier fails the whole plan closed as a `blocker`) plus the root, reads each manifest, and classifies `catalog:` / `catalog:<name>` / plain ranges. `locations` cannot be the source here because pnpm's collector omits lower-versioned workspaces. Per candidate: only plain → `pnpm -r update`; only catalog → one `CatalogEdit` per distinct referenced catalog; a mix or unreadable manifest → fail-closed blocker.
  - **Workspace monorepos (npm/pnpm/bun `workspaces`) are supported**: collect populates `Candidate.locations` (repo-relative workspace paths; root = `""`) and `pm.updatePlan(candidates, repo)` builds workspace-scoped commands from it (`npm install … -w <ws>`; a single `pnpm -r update …`; `bun add --cwd <ws> …`), so an update only ever touches the workspace(s) that already declare a dependency, never the root. bun needs a name→path bridge because `bun outdated -r` names the workspace but `bun add --cwd` wants the path — `collect.ts`'s `bunWorkspaceMap` expands the root `workspaces` globs (literal or single-level `dir/*`; deeper globs and unmapped workspace names fail closed). yarn workspaces stay out of scope.
  - **Two scope gates guard the diff (`core/scope.ts`), both fail-closed gates over a git diff.**
    - `checkBumpScope(repo, preBumpSha, members, catalogEdits)` runs on the working-tree diff against the IMMUTABLE pre-bump sha (never a ref name — the lifecycle scripts it defends against can move refs, which would let them pick the "before" content) **before the mechanical bump is committed** and ALLOW-lists only the plan's own writes in the files that WOULD enter that commit: in each changed `package.json`, every key outside the four dependency sections must be canonically identical, and inside them no key may be added/removed and a changed value is legal only for a member whose OLD value was not a `catalog:` reference (a de-catalog) and whose NEW value is EXACTLY `latest`/`^latest`/`~latest` — a whole-string grammar, deliberately not a substring/range check, so `npm:evil@<latest>` alias redirects, `git:`/`file:`/`link:`/URL specifiers, and exotic compound ranges all fail closed; in `pnpm-workspace.yaml`, only entries the plan's `catalogEdits` name, in exactly the catalog each edit targets (a default edit may land in `catalog` or `catalogs.default`; a same-named entry in an untargeted catalog may not change at all), and nothing else (any other key, added/removed entries or named catalogs, an unparseable/non-map side, creation/deletion → violation); PM lockfiles pass uninspected (generated, not executable); a new/deleted/unparseable manifest fails closed. It exists to catch a poisoned install lifecycle script rewriting a manifest during the bump — which would otherwise ride along in the "mechanical" commit, invisible to the fixer gate. It reuses `git.ts:fileAtRef` (pre-bump side) + the worktree read.
    - `checkFixScope(repo, bumpSha)` checks everything the fixer changed relative to the **bump commit** (already HEAD when the fixer starts): every `DENY`-list path (PM config files, CI config, git hooks), **any** `package.json` (by basename), **every** PM lockfile, and `pnpm-workspace.yaml` are all violations — fail-closed. There are **no guarded fields and no catalog carve-out here**: the deterministic bump already applied and committed every legitimate dependency-state change and owns it entirely, so anything the fixer touches in that space can only be scope creep. `FIXER_DENIED_FILES`'s lockfile half derives from pm.ts's `ALL_PM_LOCKFILES`, so a new PM's lockfiles extend the gate automatically; still extend `DENY` by hand for any new config/execution surface, and mirror path additions in `agents/fixer.md` so the fixer avoids them instead of burning a run on a scope violation. The gate keys its package.json check on the exact basename, so `git.ts`'s `changedPaths` must list untracked files individually (`--untracked-files=all`) — git's default collapses a brand-new directory to `newdir/`, which would let a new `newdir/package.json` slip past the gate and get committed by the catch-all `commitAll`.
    - **The pnpm catalog diff shape is gated by `checkBumpScope`, not by trusting `bump.ts` alone**: `bump.ts` owns the `pnpm-workspace.yaml` catalog move deterministically (fail-closed on a missing entry / unparseable file / non-string value; a default-catalog edit resolves to `catalog.<name>` then `catalogs.default.<name>`, a named edit to its own catalog only), AND `checkBumpScope` independently allow-lists exactly the resulting diff (the plan's own edits moving to exactly the target, nothing else) before the commit — a lifecycle script that also edited a non-catalog key or an untargeted catalog is caught there. The fixer gate still denies the whole file (the bump owns it). bun keeps catalogs in package.json and its catalogs are not updated yet (a recorded follow-up).
    - **Tests are the one surface the scope gate cannot deny** (adapting a test to a changed API is a legitimate update), so they are handled by _visibility_ instead: `core/test-changes.ts` classifies the committed diff and surfaces a ⚠️ warning in the PR body/step summary. Not a gate — do not add a deny for tests.
- **The verification gate refusing to vouch (`no-verify-scripts`) means no PR**, by design.
  - Verify steps come from script auto-detection (`build`/`lint`/`test` — `typecheck` deliberately not detected), or from the `verify_commands` input (`DEPVISOR_VERIFY_COMMANDS`, newline-separated shell commands), which **replaces** auto-detection entirely.
  - `verify_commands` must only ever come from the workflow file/env, never from the agent-writable target tree.
  - The same gate also runs on the base tip **before the agent, once per processed group** (`baseline-red` on the first — base itself is broken; `reset-failed` on a later group — the between-groups tree reset was incomplete). Either is a fail-closed run-level stop, so post-update failures are always attributable to that group's update. The reset (`resetToBase` + reinstall) uses `git clean -fd` (no `-x`), so gitignored verify artifacts can survive between groups; per-group baseline re-verification is the safety net that catches a leak as `reset-failed`.
- **`.git/` is outside the scope gate (it only sees the working tree), so target install/verification scripts are treated as able to write it. Agent repo tools reject `.git` entirely.**
  - Every git invocation disables local hooks (`NO_HOOKS = -c core.hooksPath=/dev/null`, exported from `git.ts`, reused in `github.ts`). Never add a git call that bypasses `run()`/the `git()` wrapper.
  - **Ref integrity is snapshot-verified around every untrusted target execution** (`git.ts:snapshotRefs`/`refDrift`/`restoreRefs`): group-boundary reinstall, baseline verification, bump lifecycle scripts, and both post-bump/post-fix verification phases run target code with `.git` reachable. The workflow snapshots refs before the first such execution, maintains expected refs across its own branch/commit writes, and checks every boundary including failures. The fixer/digest cannot reach `.git`; the post-digest seal remains defense in depth for delayed target processes.
  - The token-holding open-pr step runs entirely inside a fresh clone (`prepareCleanPush` in `github.ts`), because `.git/config` alone is a rich command-execution surface (`credential.helper=!cmd` fires on push auth, plus `core.fsmonitor`, `diff.external`, `filter.*`, …) that hook-disabling does not cover. Do not make the open-pr step operate on the target checkout's `.git`.
  - The push **target** is equally untrusted: in CI it comes from `DEPVISOR_REMOTE_URL` (Actions context `${server_url}/${repository}`); only when that is unset (local dev on a trusted machine) does it fall back to the checkout's `remote.origin.url` — the one read the open-pr step performs against the target checkout's `.git`. Either way it must pass `isNetworkRemote` — a push to a local/`file:`/helper remote would run that remote's server-side hooks in this `GH_TOKEN` process.
  - The payload's **`title`/`body` are untrusted at this boundary too** (written by the tokenless step): `openPrWithGh` re-sanitizes them at the exit (`sanitizePrBody`, preserving only a strictly-validated **trailing** versions marker — `extractVersionsMarker` is end-anchored, and the workflow's skip-if-up-to-date compares the same trailing marker, because a marker-shaped string can survive sanitizing inside a code span mid-body and must not be able to impersonate the marker), just as it re-validates `base` and the branch's committers.
  - The **inherited target-script environment** is untrusted too: lifecycle/verification scripts can write `$HOME/.gitconfig` or inject through runner files. All git/gh in `github.ts` therefore run via `buildSecureEnv` (token env allowlist only, clean HOME/XDG/GH config dir, `GIT_CONFIG_NOSYSTEM=1`, no inherited `GIT_*`/loader/exec env, PATH pinned, binaries by absolute path). Keep every subprocess in this module on that env.
- **Keep the Flue dependency thin**: `defineAgent` / `defineWorkflow` / `defineTool` / `defineAgentProfile` / the default virtual sandbox / `harness.session(name)` / `session.task` (structured result + named subagent + response usage/model/data) / workflow `log` / `ResultUnavailableError` / `FlueError`, exact-pinned beta. `session.prompt` and `local()` are not used.
  - `defineTool` is used for release notes and the profile-separated bounded repo bridge; `defineAgentProfile` + `session.task({ agent })` for fixer/digest; `FlueError` for digest fail-soft. Keep the rest thin so the AI-SDK escape hatch stays open.
  - All are "Flue lets you not write it" rather than "only Flue can express it," so portability is preserved.

### Development workflow

`.github/workflows/depvisor.yml` is the **development** workflow: it runs the composite action from the checkout via `uses: ./`, matching consumer workflows except for `uses: ./` and `install_command: skip` (the target is depvisor itself, a pnpm repo — the action's own `pnpm install` covers it).

The composite action has two documented GitHub-runner quirks (nested `uses:` doesn't evaluate `github.action_path`; pnpm/action-setup breaks absolute paths) — read the comments in action.yml before touching it. That second quirk is also load-bearing on purpose: `pnpm/action-setup` reads `packageManager` from `package_json_file` (default = the target checkout's `package.json`) and refuses when it differs from the pinned `version` (#35), so `package_json_file` is pointed at depvisor's own manifest — the absolute path mangles to a nonexistent file, so the target's `packageManager` is never read and the pinned `version` stands.

### Release workflow

`.github/workflows/release.yml` is depvisor's **own distribution** automation (separate from the development `depvisor.yml`; it touches no target checkout and runs no agent, so the token-separation / fresh-clone / scope-gate invariants are untouched — it only tags and releases this repo). **release-please** (manifest mode, `release-type: node`, config in `release-please-config.json` + `.release-please-manifest.json`) reads Conventional Commits on `main` and keeps one **Release PR** open that computes the next version, regenerates `CHANGELOG.md`, and bumps `package.json` `version`. **Merging that Release PR** is the human gate that cuts the `vX.Y.Z` tag + GitHub Release; the second job (`tag-major`, hand-carved shell) then force-moves the movable `v1`/`v1.x` tags to the released commit so `morinokami/depvisor@v1` in the README keeps resolving. Third-party actions are **SHA-pinned** (`ci.yml` family style). depvisor's runtime code never reads its own `package.json` `version`, so a bump is distribution-only — no behavior change (keep it that way).

- **Dependency-update commit-type convention** (release-please reads the squash subject on `main`; the repo is set to "squash message = PR title"): a dependency bump depvisor absorbs with no observable behavior change is **`chore(deps):` / `build(deps):`** — silent (hidden in `changelog-sections`, drives no release). The two exceptions that must **surface + ship**: a **security fix** in a bundled dependency is **`fix(deps):`** (a consumer runs depvisor's bundled deps on their runner, so a CVE fix changes their attack surface; `chore(deps):` would leave it unshipped), and a bump whose behavior **leaks** through is `fix:` / `feat:` **described by the behavior**, not "bump X". **Footgun**: depvisor's own PRs are titled `deps: update …` (`pr.ts`), which is silent by design — so a security update needs the merger to **retype the squash subject to `fix(deps):`**, or it never releases.
- `bootstrap-sha` pins release-please's scan start to the setup commit so it does not fold the whole `feat`/`fix` history into the first CHANGELOG; the first release is forced to `1.0.0` via a one-shot `Release-As: 1.0.0` commit footer on `main`. Do **not** add the README's `@v1` to `extra-files` — the movable pin must stay `@v1` for all of v1.x.
- A future `release: published` publish workflow (Marketplace) would not fire from a `GITHUB_TOKEN`-created release (GitHub's recursion guard); switch release-please to a PAT / GitHub App token then. Recorded, not needed today.

## Gotchas

- Env config treats **empty strings as "not set"** everywhere (`||`/falsy checks, never `??` — see `update.ts`, `agents/depvisor.ts`, `shared/target.ts`, `open-pr.ts`): the composite action forwards unset inputs as empty strings. `DEPVISOR_OPEN_PULL_REQUESTS_LIMIT` follows this too (empty → `5`, via `budget.ts`'s `parseOpenPullRequestsLimit`; a non-positive-integer value fails the run with `bad-open-pull-requests-limit`), as does `DEPVISOR_MINIMUM_RELEASE_AGE` (empty → `1`, `0` = explicit disable, via `release-age.ts`'s `parseMinimumReleaseAge`; anything else fails the run with `bad-minimum-release-age`), `DEPVISOR_MINIMUM_RELEASE_AGE_EXCLUDE` (empty → no exemptions, via `release-age.ts`'s `parseMinimumReleaseAgeExclude`; an unrecognized entry fails the run with `bad-minimum-release-age-exclude`), `DEPVISOR_IGNORE` (empty → no rules, via `ignore.ts`'s `parseIgnore`; an unrecognized entry fails the run with `bad-ignore`), and `DEPVISOR_SUGGEST_FEATURES` (empty/`"false"` → off, `"true"` → on, via `suggest-features.ts`'s `parseSuggestFeatures`; anything else fails the run with `bad-suggest-features`). `DEPVISOR_LLM_MODEL` has **no default** — the agent factory throws if it's unset/empty (action.yml fail-fasts earlier with a clearer `::error`, since GitHub doesn't enforce `required:` on action inputs).
- `npm outdated` / `pnpm outdated` read the **installed tree**: without an install in the fixture, candidates are hidden. Both exit 1 when updates exist (JSON still on stdout) — and npm reports hard failures (registry outage, broken manifest) as an `{"error": {code, summary}}` object on stdout with that same exit 1, so `collectCandidates` fails closed on that shape (and on empty output + non-zero exit) instead of reading it as a green "no updates"; a real dependency named `error` still parses because its entry carries `current`/`latest`, which npm's error object lacks. The argv carry the flags that make workspace deps visible/locatable: npm uses `--json --long` (each entry gains `type` for dev/prod and `dependedByLocation` for the workspace path; a dep at differing versions across workspaces arrives as an **array**, merged into one candidate — lowest `current`, the distinct set retained in `currents` for advisory matching, union `locations`, dev only if every occurrence is dev), pnpm uses `-r --format json` (without `-r` only the root package's deps are reported; entries carry `dependencyType` and `dependentPackages[].location`, which is name-keyed so cross-workspace occurrences collapse to a single entry whose `current` is the **highest** installed version — the update is unaffected since `pnpm -r update` reaches every workspace, but the dropped lower versions are invisible to advisory matching, above). Hence the separate parsers in `collect.ts`.
  - `bun outdated` is the odd one out: it reads the **committed lockfile** (works without an install, errors without a lockfile), always exits 0, and has **no JSON output** — `parseBunOutdated` in `collect.ts` parses its ASCII table and fails closed on any format drift. depvisor always passes `-r`, so the table always carries a `Workspace` column (the declaring workspace's **name**, root included); the parser merges a package's per-workspace rows and resolves names to paths via `bunWorkspaceMap`. `collectCandidates` strips `FORCE_COLOR` from the spawn env: it flips bun to box-drawing + ANSI even when piped, and beats `NO_COLOR` (verified on bun 1.3.14).
- `fixtures/sample-app/` (npm), `fixtures/sample-app-pnpm/`, and `fixtures/sample-app-bun/` are gitignored throwaway git repos generated from the shared `sample-app.template/` (committed; it deliberately contains code that the lru-cache major update breaks). The pnpm variant gets its own `pnpm-workspace.yaml` written by `init-fixture.sh` — without it, pnpm walks up and captures the fixture into **depvisor's own workspace**, installing nothing locally. The bun variant needs no such guard (bun anchors workspaces on package.json `workspaces`, which depvisor lacks).
  - `fixtures/sample-app-workspaces/` (also gitignored) is the npm-workspaces monorepo variant, from its **own** committed template `sample-app-workspaces.template/` (two packages, `@sample/core` + `@sample/web`, sharing `semver` so collection merges it and the update scopes `-w` to both). The template ships no lockfile, so `--workspaces` generates the single root `package-lock.json` before the baseline commit (like pnpm/bun). Root `build`/`test` scripts fan out with `npm run … --workspaces --if-present`, which is what the verify gate detects.
  - `fixtures/sample-app-bun-workspaces/` is the bun-workspaces counterpart (`--pm bun --workspaces`), from `sample-app-bun-workspaces.template/`. Same two-package shape, but the root scripts fan out with `bun run --filter='*' …` and the baseline generates `bun.lock`. `@sample/web` declares its own `@types/semver` because bun's isolated linker (unlike npm's hoisting) won't surface `@sample/core`'s copy.
  - `fixtures/sample-app-pnpm-workspaces/` is the pnpm-workspaces counterpart (`--pm pnpm --workspaces`), from `sample-app-pnpm-workspaces.template/`. Same two-package shape, root scripts fan out with `pnpm -r run …`, and the shared `semver` is **catalog-pinned** (`catalog:` in both packages, version in the template's own `pnpm-workspace.yaml` — which also serves as the workspace-capture guard the single-package pnpm fixture writes by hand), so a run against it exercises the deterministic pnpm catalog edit (`bump.ts`) end to end. Both packages declare their own `@types/semver` (pnpm's isolated linker, like bun's).
- No build step for the deterministic core: `src/core/`, `src/open-pr.ts`, `src/report-status.ts`, `src/install-target.ts`, `src/check-credentials.ts`, `src/dev/scan.ts`, and the tests run directly under Node with explicit `.ts` import extensions; tsconfig is maximally strict (`exactOptionalPropertyTypes`, `noUncheckedIndexedAccess`).
  - The **agent module is the exception**: `agents/depvisor.ts` uses Flue-bundler-only `.md` imports (`with { type: "markdown" }`, for `fixer.md` and `digest.md`), so it is loadable only under `flue run`/`flue build`, not plain `node` (`ERR_UNKNOWN_FILE_EXTENSION`).
  - Keep it that way — nothing on a plain-node path (tests, `open-pr.ts`, `report-status.ts`, `install-target.ts`, `check-credentials.ts`, `dev/scan.ts`, `core/`) may import an agent/workflow module.
- For Flue API questions, use the `flue` skill (`flue docs search/read`) rather than guessing — the dependency is a beta.
