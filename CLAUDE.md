# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

depvisor prepares Dependabot-style dependency-update PRs, shipped as a GitHub **composite action** (`action.yml`). It does what a human does to such a PR â€” apply the update, fix whatever it breaks, verify, and open the PR â€” but splits the work by trust: the dependency bump, install, and **all** verification are **deterministic**, and an LLM is used only for the two jobs that need judgment â€” fixing source breakage when the deterministic checks fail (the **fixer**), and writing the PR's reviewer digest (the **digest**). Keep `README.md` in sync when behavior changes â€” and `start.md` too: it is the agent-facing setup guide the README tells AI coding agents to fetch (by raw URL, so it must stay self-contained), and it duplicates the consumer workflow template, input names, and status names on purpose.

**Whenever you change code, check whether this file (CLAUDE.md) is still accurate â€” commands, invariants, gotchas â€” and update it in the same change if not.** Keep it lean: this file is loaded into every session, so a line Claude could learn by reading the code costs more than it is worth. Module-level detail belongs in that module's doc header; flow and release detail belong in the skills listed below.

## Commands

depvisor itself uses **pnpm** (the default fixture is an npm project; a pnpm variant exists). Node >= 22.19 required (Flue; declared in package.json `engines`).

```bash
pnpm test                                    # unit tests (node --test), no API key
node --test test/grouping.test.ts            # single test file
pnpm run typecheck && pnpm run lint && pnpm run fmt:check && pnpm run knip
                                             # CI gates (plus test) â€” all must pass
node src/dev/scan.ts fixtures/sample-app --verify   # deterministic core E2E, no API key
pnpm run fixture:init                        # create the throwaway fixture (required before agent runs)
pnpm run fixture:init:pnpm                   # pnpm variant â†’ fixtures/sample-app-pnpm (tests PM detection)
pnpm run fixture:init:bun                    # bun variant â†’ fixtures/sample-app-bun (needs a local bun)
pnpm run fixture:init:workspaces             # npm-workspaces monorepo â†’ fixtures/sample-app-workspaces
pnpm run fixture:init:workspaces:bun         # bun-workspaces monorepo â†’ fixtures/sample-app-bun-workspaces
pnpm run fixture:init:workspaces:pnpm        # pnpm-workspaces monorepo (catalog-pinned semver) â†’ fixtures/sample-app-pnpm-workspaces
pnpm exec flue run update                    # agent run; needs API key + DEPVISOR_LLM_MODEL in .env (DEPVISOR_OPEN_PULL_REQUESTS_LIMIT optional)
node src/open-pr.ts                          # push + PR from every emitted payload; needs GH_TOKEN + remote
cd fixtures/sample-app && git checkout -f main && npm install        # reset fixture to green baseline
```

Fixture agent run (scans and picks groups itself, same path as CI):
`DEPVISOR_TARGET_REPO="$PWD/fixtures/sample-app" pnpm exec flue run update`
(add `DEPVISOR_OPEN_PULL_REQUESTS_LIMIT=1` to cap the run at a single PR; the default ceiling is 5)

## Architecture

One split governs everything: **deterministic code does anything needing reproducibility, idempotency, or safety â€” the dependency bump, the install, and all verification; an LLM does only the two jobs that need judgment â€” fixing source breakage when the deterministic checks fail (the fixer), and writing the PR digest â€” each boxed between deterministic gates.**

`src/core/` is the deterministic, LLM-free, unit-tested half. `src/agents/` (+ `fixer.md` / `digest.md`) and `src/workflows/update.ts` are the only LLM-driven code â€” `src/tools/` and `flue.config.ts` touch Flue APIs but stay LLM-free. Detail lives next to what it describes:

| Where                        | What it covers                                                               |
| ---------------------------- | ---------------------------------------------------------------------------- |
| `src/core/CLAUDE.md`         | the core pipeline, per-module failure policy, shared leaves, exit boundaries |
| `src/CLAUDE.md`              | the agent capability boundary, the entrypoint/token map                      |
| each module's file header    | that module's rationale â€” **write module detail there, not here**            |
| `depvisor-update-flow` skill | the per-group loop, its gates, the status vocabulary                         |
| `depvisor-release` skill     | `action.yml`, the dev workflow, release-please, commit-type convention       |

## Invariants

Preserve these when changing the update flow (see the `depvisor-update-flow` skill). These are the lines that, if removed, would let a change quietly break the trust model.

- **Token separation is structural, not instruction-based**: the agent step holds only the LLM key; GitHub tokens exist only in the snapshot and open-pr steps, with `persist-credentials: false` on checkout. Never hand the agent step a token or let the agent run git/GitHub operations.
  - The deterministic bump (`bump.ts`) and the multi-PR loop's group-boundary reinstall (`runInstall`) both run the target's lifecycle scripts inside the token-free agent step. This adds **no new exposure class**: install scripts already run there (the pre-agent install and the between-groups reinstall), and no GitHub token is ever present in that step.
  - **Agent host isolation is capability-based**: the root uses Flue's in-memory virtual sandbox, so built-in fs/shell cannot reach the runner. `repo-files.ts` is the only host bridge; digest receives read/list/search only, fixer additionally receives bounded repo-relative write/replace/remove, and `.git`/symlink/`..` escapes are rejected. Neither role can rewrite depvisor's own checkout or the source the later GH_TOKEN step executes. Do not reintroduce `local()` or a host shell tool without restoring an OS-level isolation boundary first.
  - The user's checkout can defeat the separation from outside (actions/checkout defaults to `persist-credentials: true`), so `core/credentials.ts` fail-closes on persisted credentials in the checkout's repo-**local** git config: Authorization extraheaders (URL-scoped or not), userinfo in http(s) remote url/pushurl and in `url.*.insteadOf`/`pushInsteadOf` rewrites, `core.sshCommand`, repo-local credential helpers. The config read follows include directives (`--includes` in `localConfigEntries`) because checkout v6+ persists the token in a separate RUNNER_TEMP file referenced via `include.path` from the repo-local config. Enforced twice â€” the `check-credentials.ts` Action step, and the `persisted-credentials` preflight status covering local/non-action runs. Best-effort over known vectors, not a guarantee. Findings never include config values, and redact URL subsections from keys â€” both positions can carry the secret (`url.<https://token@â€¦>.insteadOf` puts it in the key).
- **Commit identity is split on purpose** (`git.ts:commitStaged`): the **author** is the resolvable `41898282+github-actions[bot]@users.noreply.github.com` â€” display-only, because Vercel-class Git integrations refuse to build a PR whose commit author resolves to no GitHub account (#46) â€” while the **committer** stays the unclaimable `depvisor[bot]@users.noreply.github.com` sentinel (`AGENT_EMAIL`), and both push-boundary guards in `github.ts` (foreign commits in `base..branch` â†’ `failed`, remote-tip human takeover â†’ `blocked`) key on the committer (`%ce`), never the author. Keeping the committer unchanged is also the migration guarantee: tips of PR branches created before the split carry the sentinel in both fields, so they keep refreshing. A human rebase/amend/web-UI edit always rewrites the committer â€” exactly the takeover signal â€” so never move a guard back onto the author (trivially forged via `--author`; these guards were never cryptographic â€” the sha-based gates are the hard ones) and never make the committer configurable. A configurable _author_ input would be safe now, but is deliberately deferred until someone asks.
- **The mechanical bump commit is made before any LLM runs** (`update.ts`): `deps: bump <pkgs>` is committed by deterministic code the moment the bump applies, so a reviewer is structurally guaranteed the AI wrote none of it; the `fix: adapt code to <pkgs>` commit exists only when the fixer adapted source. Install/verification leftovers never fold into it: clean-tree/worktree-drift gates reject them. Never let an agent create either commit.
- **Branch name = stable group key = PR identity**: changing grouping or naming logic changes PR identity and breaks idempotency/skip-if-up-to-date.
  - The `minimum_release_age` clamp is a **known, documented exception**: a clamped candidate's `updateType` is recomputed, so a cooldown-window major can surface first as e.g. `prod/<name>` and later, once matured, move to `major/<name>` â€” a different branch and a new PR. The stranded earlier PR is not auto-closed and keeps occupying a `open_pull_requests_limit` slot until a human closes/merges it (README documents this; a superseded-PR closer is a recorded future feature, not present).
  - **Security prioritization (`advisories.ts`) is ordering ONLY, so it is NOT an identity exception**: it reorders groups but never touches keys, membership, or the installed version. Keep it that way â€” a "promote by editing the group key" shortcut would break idempotency. Its one determinism caveat is like the clamp's: OSV data is time-varying, so "same input â†’ same order" holds only within an advisory snapshot; the sort itself is stable (`Array.prototype.sort`, Node 22) and deterministic given that snapshot.
- **Package-manager detection is pinned at preflight** (`core/pm.ts`, called once in `update.ts` preflight against the trusted base tree; also by `install-target.ts` and `dev/scan.ts`).
  - Supported: npm + pnpm + bun; yarn and ambiguous lockfile mixes fail closed (`unsupported-package-manager` / `ambiguous-package-manager`, no agent run). bun repos need the bun binary on the runner (`oven-sh/setup-bun`, version pinned â€” see README).
  - Never re-detect after the agent has run â€” lockfiles are agent-writable, so post-agent detection would let the agent switch which commands the trusted steps execute.
  - The detected `pm` drives collect, verify `run` prefixes, the deterministic bump (`pm.updatePlan`, applied by `bump.ts`), the group-boundary reinstall in the multi-PR loop (via `pm.installCommand`, when `install_command` is `auto`/`skip`), and (via `pm.lockfiles`) the lockfile half of the two-commit split's bump commit â€” `git.ts`'s `manifestBumpPaths` pairs those lockfiles with every changed `package.json` (root or workspace), matched by basename.
  - `pm.updatePlan(candidates, repo, { pinExact })` builds the deterministic `UpdatePlan` (`{ catalogEdits, commands, pinExact }`) that `bump.ts` executes; `pinExact: true` whenever the `minimum_release_age` cooldown is active. bun's plan changes under it (`bun add <name>@<latest>`, no `^`), because bun both writes the specifier verbatim and resolves ranges at install time â€” a caret range would let the install reach back into the cooldown window that the clamp just excluded. npm installs exact from `npm install <name>@<ver>` and pnpm from `pnpm -r update <name>@<ver>`, so their commands ignore the flag â€” but pnpm's **catalog entries** tighten under it (written exact, same reasoning as bun: a hand-written range is resolved fresh by the follow-up install).
  - **pnpm catalogs are moved by a deterministic YAML edit, not a pnpm command**: pnpm has no command that moves a catalog entry to a SPECIFIC version â€” `pnpm -r update <name>@<ver>` **de-catalogs** instead (rewrites each workspace's `catalog:` specifier to the plain version, leaving the catalog entry stale), and `--latest` (which does edit the catalog) rejects version specs (`ERR_PNPM_LATEST_WITH_SPEC`); both verified on pnpm 11.9. So `pnpmUpdatePlan` routes catalog-pinned members to `catalogEdits` (excluded from the update command), and `bump.ts` rewrites the `catalog`/`catalogs` entry through the `yaml` Document API, then runs `pnpm install --no-frozen-lockfile` (CI=true flips pnpm's install default to frozen, which would fail on the edited catalog). The edit is LLM-free code (its correctness is a `bump.ts` unit-test concern, not an agent-trust gate); `pm.extraBumpFiles` routes the changed pnpm-workspace.yaml into the mechanical bump commit (`git.ts:manifestBumpPaths` matches it by exact root path, not basename).
    - **Which members are catalog-pinned is decided by the actual DECLARATIONS across EVERY workspace, not a name-global scan of the catalog keys and not `Candidate.locations`**: `pnpmUpdatePlan` enumerates the workspace set itself (`pnpmWorkspaceDirs` expands pnpm-workspace.yaml's `packages` patterns â€” literals, `*`, `**`, `!negations`, with leading `./` normalized to pnpm's equivalent repo-relative spelling; anything fancier fails the whole plan closed as a `blocker`) plus the root, reads each manifest, and classifies `catalog:` / `catalog:<name>` / plain ranges. `locations` cannot be the source here because pnpm's collector omits lower-versioned workspaces. Per candidate: only plain â†’ `pnpm -r update`; only catalog â†’ one `CatalogEdit` per distinct referenced catalog; a mix or unreadable manifest â†’ fail-closed blocker.
  - **Workspace monorepos (npm/pnpm/bun `workspaces`) are supported**: collect populates `Candidate.locations` (repo-relative workspace paths; root = `""`) and `pm.updatePlan(candidates, repo)` builds workspace-scoped commands from it (`npm install â€¦ -w <ws>`; a single `pnpm -r update â€¦`; `bun add --cwd <ws> â€¦`), so an update only ever touches the workspace(s) that already declare a dependency, never the root. bun needs a nameâ†’path bridge because `bun outdated -r` names the workspace but `bun add --cwd` wants the path â€” `collect.ts`'s `bunWorkspaceMap` expands the root `workspaces` globs (literal or single-level `dir/*`; deeper globs and unmapped workspace names fail closed). yarn workspaces stay out of scope.
  - **Two scope gates guard the diff (`core/scope.ts`), both fail-closed gates over a git diff.**
    - `checkBumpScope(repo, preBumpSha, members, catalogEdits)` runs on the working-tree diff against the IMMUTABLE pre-bump sha (never a ref name â€” the lifecycle scripts it defends against can move refs, which would let them pick the "before" content) **before the mechanical bump is committed** and ALLOW-lists only the plan's own writes in the files that WOULD enter that commit: in each changed `package.json`, every key outside the four dependency sections must be canonically identical, and inside them no key may be added/removed and a changed value is legal only for a member whose OLD value was not a `catalog:` reference (a de-catalog) and whose NEW value is EXACTLY `latest`/`^latest`/`~latest` â€” a whole-string grammar, deliberately not a substring/range check, so `npm:evil@<latest>` alias redirects, `git:`/`file:`/`link:`/URL specifiers, and exotic compound ranges all fail closed; in `pnpm-workspace.yaml`, only entries the plan's `catalogEdits` name, in exactly the catalog each edit targets (a default edit may land in `catalog` or `catalogs.default`; a same-named entry in an untargeted catalog may not change at all), and nothing else (any other key, added/removed entries or named catalogs, an unparseable/non-map side, creation/deletion â†’ violation); PM lockfiles pass uninspected (generated, not executable); a new/deleted/unparseable manifest fails closed. It exists to catch a poisoned install lifecycle script rewriting a manifest during the bump â€” which would otherwise ride along in the "mechanical" commit, invisible to the fixer gate. It reuses `git.ts:fileAtRef` (pre-bump side) + the worktree read.
    - `checkFixScope(repo, bumpSha)` checks everything the fixer changed relative to the **bump commit** (already HEAD when the fixer starts): every `DENY`-list path (PM config files, CI config, git hooks), **any** `package.json` (by basename), **every** PM lockfile, and `pnpm-workspace.yaml` are all violations â€” fail-closed. There are **no guarded fields and no catalog carve-out here**: the deterministic bump already applied and committed every legitimate dependency-state change and owns it entirely, so anything the fixer touches in that space can only be scope creep. `FIXER_DENIED_FILES`'s lockfile half derives from pm.ts's `ALL_PM_LOCKFILES`, so a new PM's lockfiles extend the gate automatically; still extend `DENY` by hand for any new config/execution surface, and mirror path additions in `agents/fixer.md` so the fixer avoids them instead of burning a run on a scope violation. The gate keys its package.json check on the exact basename, so `git.ts`'s `changedPaths` must list untracked files individually (`--untracked-files=all`) â€” git's default collapses a brand-new directory to `newdir/`, which would let a new `newdir/package.json` slip past the gate and get committed by the catch-all `commitAll`.
    - **The pnpm catalog diff shape is gated by `checkBumpScope`, not by trusting `bump.ts` alone**: `bump.ts` owns the `pnpm-workspace.yaml` catalog move deterministically (fail-closed on a missing entry / unparseable file / non-string value; a default-catalog edit resolves to `catalog.<name>` then `catalogs.default.<name>`, a named edit to its own catalog only), AND `checkBumpScope` independently allow-lists exactly the resulting diff (the plan's own edits moving to exactly the target, nothing else) before the commit â€” a lifecycle script that also edited a non-catalog key or an untargeted catalog is caught there. The fixer gate still denies the whole file (the bump owns it). bun keeps catalogs in package.json and its catalogs are not updated yet (a recorded follow-up).
    - **Tests are the one surface the scope gate cannot deny** (adapting a test to a changed API is a legitimate update), so they are handled by _visibility_ instead: `core/test-changes.ts` classifies the committed diff and surfaces a âš ď¸Ź warning in the PR body/step summary. Not a gate â€” do not add a deny for tests.
- **The verification gate refusing to vouch (`no-verify-scripts`) means no PR**, by design.
  - Verify steps come from script auto-detection (`build`/`lint`/`test` â€” `typecheck` deliberately not detected), or from the `verify_commands` input (`DEPVISOR_VERIFY_COMMANDS`, newline-separated shell commands), which **replaces** auto-detection entirely.
  - `verify_commands` must only ever come from the workflow file/env, never from the agent-writable target tree.
  - The same gate also runs on the base tip **before the agent, once per processed group** (`baseline-red` on the first â€” base itself is broken; `reset-failed` on a later group â€” the between-groups tree reset was incomplete). Either is a fail-closed run-level stop, so post-update failures are always attributable to that group's update. The reset (`resetToBase` + reinstall) uses `git clean -fd` (no `-x`), so gitignored verify artifacts can survive between groups; per-group baseline re-verification is the safety net that catches a leak as `reset-failed`.
- **`.git/` is outside the scope gate (it only sees the working tree), so target install/verification scripts are treated as able to write it. Agent repo tools reject `.git` entirely.**
  - Every git invocation disables local hooks (`NO_HOOKS = -c core.hooksPath=/dev/null`, exported from `git.ts`, reused in `github.ts`). Never add a git call that bypasses `run()`/the `git()` wrapper.
  - **Ref integrity is snapshot-verified around every untrusted target execution** (`git.ts:snapshotRefs`/`refDrift`/`restoreRefs`): group-boundary reinstall, baseline verification, bump lifecycle scripts, and both post-bump/post-fix verification phases run target code with `.git` reachable. The workflow snapshots refs before the first such execution, maintains expected refs across its own branch/commit writes, and checks every boundary including failures. The fixer/digest cannot reach `.git`; the post-digest seal remains defense in depth for delayed target processes.
  - The token-holding open-pr step runs entirely inside a fresh clone (`prepareCleanPush` in `github.ts`), because `.git/config` alone is a rich command-execution surface (`credential.helper=!cmd` fires on push auth, plus `core.fsmonitor`, `diff.external`, `filter.*`, â€¦) that hook-disabling does not cover. Do not make the open-pr step operate on the target checkout's `.git`.
  - The push **target** is equally untrusted: in CI it comes from `DEPVISOR_REMOTE_URL` (Actions context `${server_url}/${repository}`); only when that is unset (local dev on a trusted machine) does it fall back to the checkout's `remote.origin.url` â€” the one read the open-pr step performs against the target checkout's `.git`. Either way it must pass `isNetworkRemote` â€” a push to a local/`file:`/helper remote would run that remote's server-side hooks in this `GH_TOKEN` process.
  - The payload's **`title`/`body` are untrusted at this boundary too** (written by the tokenless step): `openPrWithGh` re-sanitizes them at the exit (`sanitizePrBody`, preserving only a strictly-validated **trailing** versions marker â€” `extractVersionsMarker` is end-anchored, and the workflow's skip-if-up-to-date compares the same trailing marker, because a marker-shaped string can survive sanitizing inside a code span mid-body and must not be able to impersonate the marker), just as it re-validates `base` and the branch's committers.
  - The **inherited target-script environment** is untrusted too: lifecycle/verification scripts can write `$HOME/.gitconfig` or inject through runner files. All git/gh in `github.ts` therefore run via `buildSecureEnv` (token env allowlist only, clean HOME/XDG/GH config dir, `GIT_CONFIG_NOSYSTEM=1`, no inherited `GIT_*`/loader/exec env, PATH pinned, binaries by absolute path). Keep every subprocess in this module on that env.
- **Keep the Flue dependency thin**: `defineAgent` / `defineWorkflow` / `defineTool` / `defineAgentProfile` / the default virtual sandbox / `harness.session(name)` / `session.task` (structured result + named subagent + response usage/model/data) / workflow `log` / `ResultUnavailableError` / `FlueError`, exact-pinned beta. `session.prompt` and `local()` are not used.
  - `defineTool` is used for release notes and the profile-separated bounded repo bridge; `defineAgentProfile` + `session.task({ agent })` for fixer/digest; `FlueError` for digest fail-soft. Keep the rest thin so the AI-SDK escape hatch stays open.
  - All are "Flue lets you not write it" rather than "only Flue can express it," so portability is preserved.

## Gotchas

- Env config treats **empty strings as "not set"** everywhere (`||`/falsy checks, never `??` â€” see `update.ts`, `agents/depvisor.ts`, `shared/target.ts`, `open-pr.ts`): the composite action forwards unset inputs as empty strings. `DEPVISOR_OPEN_PULL_REQUESTS_LIMIT` follows this too (empty â†’ `5`, via `budget.ts`'s `parseOpenPullRequestsLimit`; a non-positive-integer value fails the run with `bad-open-pull-requests-limit`), as does `DEPVISOR_MINIMUM_RELEASE_AGE` (empty â†’ `1`, `0` = explicit disable, via `release-age.ts`'s `parseMinimumReleaseAge`; anything else fails the run with `bad-minimum-release-age`), `DEPVISOR_MINIMUM_RELEASE_AGE_EXCLUDE` (empty â†’ no exemptions, via `release-age.ts`'s `parseMinimumReleaseAgeExclude`; an unrecognized entry fails the run with `bad-minimum-release-age-exclude`), `DEPVISOR_IGNORE` (empty â†’ no rules, via `ignore.ts`'s `parseIgnore`; an unrecognized entry fails the run with `bad-ignore`), and `DEPVISOR_SUGGEST_FEATURES` (empty/`"false"` â†’ off, `"true"` â†’ on, via `suggest-features.ts`'s `parseSuggestFeatures`; anything else fails the run with `bad-suggest-features`). `DEPVISOR_LLM_MODEL` has **no default** â€” the agent factory throws if it's unset/empty (action.yml fail-fasts earlier with a clearer `::error`, since GitHub doesn't enforce `required:` on action inputs).
- `npm outdated` / `pnpm outdated` read the **installed tree**: without an install in the fixture, candidates are hidden. Both exit 1 when updates exist (JSON still on stdout) â€” and npm reports hard failures (registry outage, broken manifest) as an `{"error": {code, summary}}` object on stdout with that same exit 1, so `collectCandidates` fails closed on that shape (and on empty output + non-zero exit) instead of reading it as a green "no updates"; a real dependency named `error` still parses because its entry carries `current`/`latest`, which npm's error object lacks. The argv carry the flags that make workspace deps visible/locatable: npm uses `--json --long` (each entry gains `type` for dev/prod and `dependedByLocation` for the workspace path; a dep at differing versions across workspaces arrives as an **array**, merged into one candidate â€” lowest `current`, the distinct set retained in `currents` for advisory matching, union `locations`, dev only if every occurrence is dev), pnpm uses `-r --format json` (without `-r` only the root package's deps are reported; entries carry `dependencyType` and `dependentPackages[].location`, which is name-keyed so cross-workspace occurrences collapse to a single entry whose `current` is the **highest** installed version â€” the update is unaffected since `pnpm -r update` reaches every workspace, but the dropped lower versions are invisible to advisory matching, above). Hence the separate parsers in `collect.ts`.
  - `bun outdated` is the odd one out: it reads the **committed lockfile** (works without an install, errors without a lockfile), always exits 0, and has **no JSON output** â€” `parseBunOutdated` in `collect.ts` parses its ASCII table and fails closed on any format drift. depvisor always passes `-r`, so the table always carries a `Workspace` column (the declaring workspace's **name**, root included); the parser merges a package's per-workspace rows and resolves names to paths via `bunWorkspaceMap`. `collectCandidates` strips `FORCE_COLOR` from the spawn env: it flips bun to box-drawing + ANSI even when piped, and beats `NO_COLOR` (verified on bun 1.3.14).
- `fixtures/sample-app/` (npm), `fixtures/sample-app-pnpm/`, and `fixtures/sample-app-bun/` are gitignored throwaway git repos generated from the shared `sample-app.template/` (committed; it deliberately contains code that the lru-cache major update breaks). The pnpm variant gets its own `pnpm-workspace.yaml` written by `init-fixture.sh` â€” without it, pnpm walks up and captures the fixture into **depvisor's own workspace**, installing nothing locally. The bun variant needs no such guard (bun anchors workspaces on package.json `workspaces`, which depvisor lacks).
  - `fixtures/sample-app-workspaces/` (also gitignored) is the npm-workspaces monorepo variant, from its **own** committed template `sample-app-workspaces.template/` (two packages, `@sample/core` + `@sample/web`, sharing `semver` so collection merges it and the update scopes `-w` to both). The template ships no lockfile, so `--workspaces` generates the single root `package-lock.json` before the baseline commit (like pnpm/bun). Root `build`/`test` scripts fan out with `npm run â€¦ --workspaces --if-present`, which is what the verify gate detects.
  - `fixtures/sample-app-bun-workspaces/` is the bun-workspaces counterpart (`--pm bun --workspaces`), from `sample-app-bun-workspaces.template/`. Same two-package shape, but the root scripts fan out with `bun run --filter='*' â€¦` and the baseline generates `bun.lock`. `@sample/web` declares its own `@types/semver` because bun's isolated linker (unlike npm's hoisting) won't surface `@sample/core`'s copy.
  - `fixtures/sample-app-pnpm-workspaces/` is the pnpm-workspaces counterpart (`--pm pnpm --workspaces`), from `sample-app-pnpm-workspaces.template/`. Same two-package shape, root scripts fan out with `pnpm -r run â€¦`, and the shared `semver` is **catalog-pinned** (`catalog:` in both packages, version in the template's own `pnpm-workspace.yaml` â€” which also serves as the workspace-capture guard the single-package pnpm fixture writes by hand), so a run against it exercises the deterministic pnpm catalog edit (`bump.ts`) end to end. Both packages declare their own `@types/semver` (pnpm's isolated linker, like bun's).
- No build step for the deterministic core: `src/core/`, `src/open-pr.ts`, `src/report-status.ts`, `src/install-target.ts`, `src/check-credentials.ts`, `src/dev/scan.ts`, and the tests run directly under Node with explicit `.ts` import extensions; tsconfig is maximally strict (`exactOptionalPropertyTypes`, `noUncheckedIndexedAccess`).
  - The **agent module is the exception**: `agents/depvisor.ts` uses Flue-bundler-only `.md` imports (`with { type: "markdown" }`, for `fixer.md` and `digest.md`), so it is loadable only under `flue run`/`flue build`, not plain `node` (`ERR_UNKNOWN_FILE_EXTENSION`).
  - Keep it that way â€” nothing on a plain-node path (tests, `open-pr.ts`, `report-status.ts`, `install-target.ts`, `check-credentials.ts`, `dev/scan.ts`, `core/`) may import an agent/workflow module.
- For Flue API questions, use the `flue` skill (`flue docs search/read`) rather than guessing â€” the dependency is a beta.
