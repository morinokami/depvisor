# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

depvisor is an AI agent, shipped as a GitHub **composite action** (`action.yml`), that does what a human does to a Dependabot-style dependency-update PR: investigate, update, fix breakage, verify, and open the PR. Keep `README.md` in sync when behavior changes.

**Whenever you change code, check whether this file (CLAUDE.md) is still accurate — commands, architecture, invariants, gotchas — and update it in the same change if not.**

## Commands

depvisor itself uses **pnpm** (the default fixture is an npm project; a pnpm variant exists). Node >= 22.19 required (Flue; declared in package.json `engines`).

```bash
pnpm test                                    # unit tests (node --test), no API key
node --test test/grouping.test.ts            # single test file
pnpm run typecheck && pnpm run lint && pnpm run fmt:check && pnpm run knip
                                             # CI gates (plus test) — all must pass
node src/dev/scan.ts fixtures/sample-app --verify   # deterministic core E2E, no API key
pnpm run fixture:init                        # create the throwaway fixture (required before agent runs)
pnpm run fixture:init:pnpm                   # pnpm variant → fixtures/sample-app-pnpm (tests PM detection)
pnpm run fixture:init:bun                    # bun variant → fixtures/sample-app-bun (needs a local bun)
pnpm run fixture:init:workspaces             # npm-workspaces monorepo → fixtures/sample-app-workspaces
pnpm run fixture:init:workspaces:bun         # bun-workspaces monorepo → fixtures/sample-app-bun-workspaces
pnpm exec flue run update                    # agent run; needs API key + DEPVISOR_MODEL in .env (DEPVISOR_MAX_OPEN_PRS optional)
node src/open-pr.ts                          # push + PR from every emitted payload; needs GH_TOKEN + remote
cd fixtures/sample-app && git checkout -f main && npm install        # reset fixture to green baseline
```

Fixture agent run (scans and picks groups itself, same path as CI):
`DEPVISOR_TARGET_REPO="$PWD/fixtures/sample-app" pnpm exec flue run update`
(add `DEPVISOR_MAX_OPEN_PRS=2` to prepare more than one PR in a single run)

## Architecture

One split governs everything: **deterministic code does anything needing reproducibility, idempotency, or safety; the agent only does investigate/update/fix/verify-loop work, boxed between deterministic gates.**

### Modules

- `src/core/` — deterministic, LLM-free, unit-tested: pm (package-manager detection + per-PM command table) → collect (npm/pnpm outdated) → grouping (stable keys) → verify gate → scope gate → git (two-commit split) → pr (payload/sanitize/labels) → github (push + `gh pr create` + labels).
  - `changelog.ts` is also here: the deterministic backing for the agent's bounded `fetch_release_notes` tool. Fixed endpoints (npm registry + GitHub Releases); it never throws, returning an "unavailable" note instead.
  - `parseGithubSlug` (same file) is what the workflow uses to attach releases/compare links to the PR body — it reads the `repository` field of the packument the run already fetched (`resolveSourceRepo`, also here, is only the release-notes lookup's internal first step nowadays). Fail-soft: unresolved → no links. `pr.ts` charset-validates every URL part at the embed boundary because the sanitizer leaves markdown links alone.
  - `version-core.ts` is the tiny leaf every version-ordering module shares: the x.y.z `Triple`, `compareTriple`, and the loose (unanchored) `parseVersionCore`. depvisor deliberately carries no semver library, and each consumer anchors its own parse variant differently on purpose — the module header documents the four flavors (loose for registry/`outdated` strings; release-age's fully-anchored `parseStable`; advisories' start-anchored OSV parse with the `"0"` sentinel; changelog's end-anchored release-tag parse).
  - `status.ts` is the deterministic status/summary boundary for Actions log UX. Its `RunStatus` is **run-level + a `groups[]` array** (one entry per group the run touched, since a run can prepare several PRs). `update.ts` writes `pr-preview/status.json` incrementally — after every group, under the red `in-progress` run status that only a graceful finish upgrades to `completed` — so a mid-loop crash leaves a status that is consistent with the payloads emitted so far AND fails the job instead of impersonating a green run; `report-status.ts` turns it into a run-level annotation plus a per-failing-group annotation and `$GITHUB_STEP_SUMMARY` Markdown. `runFailsJob` fails the job when the run-level status OR any group status is non-benign. `RUN_OUTPUT_SCHEMA`/`toRunOutput` (same file) single-source the workflow's `output` shape — a `RunStatus` projection that drops per-group `packages`/`prUrl` — so the Flue output cannot drift from the status file. `RUN_STATUS_FILE` lives in the tiny `status-file.ts` leaf so `pr.ts` (which clears prior output) and `status.ts` avoid an import cycle.
    - `GroupResult.usage` (a `GroupUsage`: token counts + `costUsd` + `model`) is the deterministic **LLM-usage visibility** (display-only, unit-tested): it is a **structural projection of Flue's `PromptResultResponse.usage`/`.model`** written by `update.ts` right after the prompt returns (so core stays Flue-free — the mapping is a plain object literal in the workflow, no Flue-type import), and is absent for outcomes that never ran the agent (skips/hold-backs/`branch-collision`/`release-age-unavailable`) or where the prompt threw before returning a response (the `ResultUnavailableError` half of `no-structured-result`). Captured **the moment the prompt returns — before both the defensive re-parse and the verdict branch**, so a `defer` (which still burned tokens) reports its usage, and so does the other `no-structured-result` half (a returned response whose re-parse rejected — tokens were spent). `sumGroupUsage` totals the groups whose agent ran (null when none did — callers render nothing rather than a misleading zero row); `renderStepSummary` shows a run-total row in the header plus a per-group row, `runLogLine` a compact `tokens=/cost=`. `costUsd` is Flue's provider-priced **estimate** (BYOK-approximate, rendered `est. ~$…`), not an invoice. **Record-only, exactly like `testChanges`**: `parseGroup` preserves it across the open-pr read→rewrite round-trip and it is deliberately **NOT** in `RUN_OUTPUT_SCHEMA` (numbers only, so no token-separation/gate impact — it is written inside the token-free agent step).
  - `budget.ts` is the deterministic `max_open_prs` policy (LLM-free, unit-tested): `parseMaxOpenPrs`, `countOpenDepvisorPrs`, and `classifyGroup` (skip-up-to-date / refresh / open-new / held-back). `max_open_prs` is a **ceiling on open depvisor PRs** (Dependabot's open-pull-requests-limit model), not a per-run throughput cap — a run opens new PRs only up to the ceiling but always refreshes existing ones, and only a newly opened PR consumes a slot.
  - `release-age.ts` is the deterministic `minimum_release_age` cooldown (LLM-free, unit-tested): `parseMinReleaseAge` (empty → `1`, `0` = explicit disable), `parseMinReleaseAgeExclude` (newline-separated bare names, blank/`#`-comment lines skipped, fail-closed `bad-min-release-age-exclude` — validated even when the cooldown is disabled so a typo fails now, not on re-enable), `fetchPackument` (fixed npm-registry endpoint, injectable fetch + timeout), and `applyReleaseAge`, which — between collect and grouping — rounds each candidate's `latest` down to the newest **stable** version that is both newer than `current` and at least N days old (publish times from packument `time` ∩ `versions`, so unpublished versions and `created`/`modified` never count; a prerelease `latest` passes only via its exact-string maturity). It recomputes `updateType` after a clamp because grouping (branch/PR identity) depends on it. **Fail-closed, unlike changelog.ts**: 404 and network error alike drop the candidate (`release-age-unavailable`, red) — this module is a defense, not display. The per-package escape hatch is `minimum_release_age_exclude` (pnpm's `minimumReleaseAgeExclude`): names in `opts.exclude` skip the fetch and the age check entirely and pass through verbatim (recorded in the result's `excluded`, green — a human opt-out for private-registry packages npmjs cannot vouch for, not a fail-open path; `heldBack` is the not-yet-mature drop). Their packument is still fetched fail-soft at PR time for links/license. The packument cache it fills is reused for the PR body's source-repo links (no second registry hit per package).
  - `ignore.ts` is the deterministic `ignore` policy (LLM-free, unit-tested): `parseIgnore` (a `{ ok; rules } | { ok:false; invalid }` union — blank lines and full-line `#` comments skipped, every other line must be `name` or `name@<major>`, else fail-closed with the offending entries) and `applyIgnore` (a pure prepend filter splitting candidates into kept/ignored, plus `describeIgnore` for the log line). Syntax is deliberately semver-library-free: `name` drops the package always, `name@<major>` only when the candidate's `latest` major matches. It runs **first — before the cooldown clamp and grouping** (see the flow below), so an ignored package costs no packument fetch / agent run / spurious `release-age-unavailable`; the trade-off is that `name@<major>` matches the raw registry `latest`, not a clamped version (conservative — documented in the module header and README). Trust model = `verify_commands`: workflow config only, never the agent-writable target tree.
  - `advisories.ts` is the deterministic **security prioritization** (LLM-free, unit-tested): `fetchAdvisories` queries OSV.dev (two fixed endpoints — a `/v1/querybatch` triage of every candidate at EACH declared `current` version, then `/v1/query` per flagged (name, current) pair for full `affected` ranges), `resolvesAdvisory` decides whether an update RESOLVES an advisory (current affected AND target not — SEMVER ranges matched by x.y.z core comparison, no semver library), `prioritizeGroups` stable-promotes resolving groups to the front, and `describeAdvisories` is the log line. It probes **every** workspace-current (`Candidate.currents`), not just the merged lowest `current`, so a vulnerability carried only by a higher-versioned workspace is not hidden behind the lowest — for **npm/bun**, whose collectors report every occurrence. **pnpm is the exception**: `pnpm outdated` reports only the highest installed version and omits lower-versioned workspaces, so `currents` there is just that highest, and an advisory affecting solely an omitted lower version is missed (fail-soft: the update still fixes every workspace; only the promotion/Security hint is lost — reconstructing pnpm's per-workspace versions via `pnpm list -r` is a deliberate non-goal). **Ordering only** — it never changes which version installs, only the processing order (so branch/PR identity is untouched), and it feeds the PR body's Security column. Promotion and display are both keyed on the advisory's **GHSA id** (from OSV `id` or `aliases`, validated by `GHSA_RE`, which `pr.ts` re-validates at the embed boundary), so the unit that promotes a group is exactly the id the PR can link; a GHSA-less advisory is not counted. **Fail-soft, the deliberate opposite of `release-age.ts`**: it is an optimization, not a defense, so any OSV failure degrades to the neutral order (`ok:false`, logged, and `ADVISORIES_UNAVAILABLE_NOTE` is appended to the completed run's summary — the run stays green, so that note is the only user-visible trace of the degradation) — including a full-query failure for a package the triage already flagged, since ordering on a partial snapshot would silently mis-rank the `max_open_prs` slots — and any unevaluable range fails toward "still affected" (no promotion on a maybe). Runs against the **post-clamp** `latest`, so a fix still inside the cooldown window does not count (cooldown wins over urgency).
  - `test-changes.ts` is the deterministic **test-change visibility** (LLM-free, unit-tested), the display-only counterpart to `scope.ts`: `isTestPath` classifies a repo path against a **static** world-convention set (`test/`/`tests/`/`__tests__/`/`__mocks__/`/`spec/`/`e2e/`/`cypress/`/`playwright/` dirs, `*.test.*`/`*.spec.*`/`*_test.*` names — deliberately NOT the target's agent-writable test-runner config), `classifyTestChanges` filters a `diffNumstat` result to the test subset, and `formatNumstatLines` renders the delta (`binary` for null counts). The scope gate **cannot deny tests** (adapting a test to a changed API is a legitimate update, indistinguishable from a poisoned agent weakening an assertion), so instead the workflow classifies the committed `base..HEAD` diff and, when tests changed, adds a ⚠️ warning section to the PR body and step summary — visibility, never a gate (**no new failing status**, always green). Completeness is a non-goal; the wording never implies an empty section proves no test was touched. `git.ts:diffNumstat` is the deterministic backing (parses `git diff --numstat --no-renames -z`; binary `-` → null counts). It uses **`--no-renames` on purpose**: git's default rename detection would collapse a moved test (`test/a.test.ts` → `src/a.ts`) into one record keyed on the non-test destination, hiding that a test was dropped out of the test globs (silently disabled while verification still passes); `--no-renames` decomposes the move into a delete of the old test path plus an add, so the test path always surfaces. Paths are charset-validated (`pr.ts:isDisplayablePath`) before embedding — filenames can hold backticks/spaces (observed in real `git diff`) that would break out of a markdown code span — with any dropped path still counted.
  - `license.ts` is the deterministic **license-change visibility** (LLM-free, unit-tested), a display-only sibling of `test-changes.ts`: `versionLicense` reads one published version's `license` field from a `release-age.ts` `Packument` as a **plain string** (the deprecated `{ type }` object form, the ancient `licenses` array, a missing/blank field, and an absent version all read as `null` = unknown), `classifyLicenseChanges` emits one `LicenseChange` per DISTINCT `from → to` license pair a member crosses — it checks the target against EVERY declared `Candidate.currents` (mirroring `advisories.ts`), not just the merged lowest `current`, so a relicense carried only by a higher-versioned workspace is not hidden; a member can therefore yield several rows, with repeated `from → to` deduped (the pnpm per-workspace-collapse caveat is the same as advisory matching). Only a KNOWN string license on both sides counts (exact string compare, no SPDX parsing — the "never interpret" policy; an unknown side is skipped so an object-form "MIT" vs a string "MIT" never manufactures a phantom change), and `describeLicenseChanges` is the log line — which control-sanitizes the registry license strings (`\p{Cc}` runs collapsed to spaces, length-capped) because the CI log is a raw stream where an embedded newline could forge an Actions `::command` (the log-boundary counterpart of `LICENSE_RE`). The workflow renders a ⚠️ warning section in the PR body (`pr.ts:licenseChangesSection`); the license strings are registry-derived = untrusted, so they are charset-gated by `pr.ts`'s `LICENSE_RE` (SPDX-shaped `[A-Za-z0-9 .+()-]`, so a backtick/`[`/`|` cannot break out of the code span or table row) with any dropped change still counted. **Fail-open, the deliberate opposite of `release-age.ts`**: display, not a defense, so an unresolvable license shows nothing rather than blocking a PR. It reuses the packument the cooldown/source-repo links already fetched (**no extra registry hit**; when the cooldown is disabled `update.ts` fetches each member's full packument once — the same round-trip the old `resolveSourceRepo` made — and derives both the source slug and the license from it). PR-body-only today (unlike `test-changes.ts`, it is NOT surfaced in the step summary — hence `isDisplayableLicense` is unexported, mirror the test-change path if that changes).
  - `install.ts` (`runInstall`) is the shared install spawn used by both `install-target.ts` (pre-agent) and the group-boundary reset in `update.ts`. It carries a 15-minute timeout (mirroring verify's per-step timeout) so a hung install cannot eat the CI job.
  - `pr.ts:deriveLabels` is the deterministic **PR-label policy** (LLM-free, unit-tested): `depvisor` always, the group's top `semver:<patch|minor|major>` (majors are isolated per package by grouping, so a group mixes at most minor+patch), `security` when a member resolves an advisory (same `advisories.resolvedByPackage` map that feeds the Security column), `dev-dependencies` when every member is a dev dep. The set rides in `PrPayload.labels` (written by the tokenless step) and is applied **only** in the token-holding step (`github.ts:applyLabels`), which first re-validates it against a fixed allowlist (`pr.ts:sanitizeLabels`, the label counterpart of `sanitizePrBody` — drops anything off-vocabulary, incl. flag-shaped/injection strings, dedupes, stabilizes). Application is **fail-soft and decoupled from PR creation**: it runs AFTER `gh pr create` (labels are NEVER passed to create — `gh pr create --label <unknown>` aborts creation, turning a missing label into a lost PR), ensuring each label with `gh label create` (no `--force`, so a user's same-named label keeps its color/description) then adding it via `gh pr edit --add-label` **one label per edit**, on both the created and refreshed paths. Per-label (not a single batched `--add-label`) is deliberate: `gh pr edit` resolves all requested labels up front and errors on any unknown one, so a batch is all-or-nothing and one label a transient create failure left missing would drop even the labels that exist. Both creating and applying a label need only `pull-requests: write` — GitHub's label endpoint accepts `issues` OR `pull-requests` write (verified empirically against the standard `GITHUB_TOKEN`), so **no `issues: write` is required**; each label is applied independently and an unappliable one is logged and skipped, never fatal. Labels never touch branch/group keys, membership, or the installed version, so PR identity is untouched (like `advisories.ts`, ordering/display only).
- `src/agents/updater.ts` (+ `updater.md` instructions) and `src/workflows/update.ts` — the only LLM-driven code (`src/tools/` and `flue.config.ts` also touch Flue APIs, but stay LLM-free).
  - The `defineAgent` updater lives in `src/agents/` per Flue's discovery convention (filename = agent name); it exports no `route`, so it is never HTTP-exposed. Its instructions are imported from `updater.md` (`with { type: "markdown" }`, Flue's convention).
  - The agent carries one bounded tool, `fetch_release_notes` (`src/tools/release-notes.ts` — `defineTool`, backed by `core/changelog.ts`) — the single narrow door for untrusted external text, never raw agent HTTP.
  - Tool definitions live in `src/tools/` (not auto-discovered by Flue; imported and attached via `tools:`).
- `src/shared/target.ts` — the one value both the agent (its `cwd`) and the workflow share: `REPO`, the target checkout in CI / throwaway fixture locally.
- `src/open-pr.ts` — the deterministic, token-holding entrypoint: it enumerates **every** payload under `pr-preview/payloads/` (filename = processing order) and pushes + `gh pr create`s each via `openPrWithGh`, patching that group's status entry (matched by branch) with the PR URL, a blocked-takeover note (`open-pr-blocked`, stays green), or an open-pr failure (`open-pr-failed`). One payload's failure does not stop the rest; it exits 1 if any failed. The ONLY command that needs `GH_TOKEN`; in CI it runs as its own Action step. `openPrWithGh` is per-payload self-contained (fresh clone + authorization checks + deterministic label application), so multi-PR is just calling it N times — do not add cross-payload state to it.
- `src/report-status.ts` — the deterministic Actions log UX entrypoint. It reads `pr-preview/status.json`, emits a run-level annotation (level from `runFailsJob`) plus one error annotation per failing group, appends a per-group Markdown step summary, and exits non-zero via `runFailsJob`.
- `src/check-credentials.ts` — the plain-node entrypoint for the credentials gate: fails the job with `::error` when `core/credentials.ts` finds persisted credentials in the target checkout. Runs as the first target-touching Action step — **before** the target install (whose lifecycle scripts must not see a token either) and regardless of `install_command`.
- `src/install-target.ts` — the plain-node entrypoint behind `install_command: auto`: detects the target's PM and runs the matching lockfile-faithful install (`npm ci` / `pnpm install --frozen-lockfile`) before the agent step.
  - It fails the job with a clear `::error` for unsupported/ambiguous PMs AND for repos without a committed lockfile — a bare `npm install` would create the lockfile and dirty the pre-agent tree, which preflight would later reject as an illegible `dirty-tree`.
- `src/dev/scan.ts` — a developer-only tool (no key) that runs detect → collect → group → verify against a repo and prints it. NOT part of the CI/composite-action flow, hence its home under `dev/`. `--min-release-age=<days>` opt-ins the cooldown clamp (hits the real npm registry; the default scan stays offline).

### The update flow

The workflow (`flue run update`) imports the updater agent and drives it. Preflight, PM detection, and collect+group run **once**; then it **loops over groups** up to the `max_open_prs` open-PR ceiling (`budget.ts`):

0. Preflight (repo root, credentials, clean tree, PM detection, base, verify steps), `parseMaxOpenPrs`, `parseMinReleaseAge`, `parseMinReleaseAgeExclude`, and `parseIgnore` — any failure is a run-level stop, no loop.
1. Collect, then the `ignore` filter (`ignore.ts`, when rules are set), then the `minimum_release_age` clamp (`release-age.ts`, when N > 0), then grouping. Ignore runs **first** so an ignored package never costs a packument fetch, an agent run, or a red `release-age-unavailable` — its `name@<major>` therefore matches the raw registry `latest` (conservative; documented). The clamp must sit between ignore and grouping because it can reclassify `updateType`, which the group key (branch/PR identity) depends on. Candidates the registry cannot vouch for are dropped here and later recorded as `release-age-unavailable` group entries (branch/group null, red); ignore drops, cooldown clamps, and not-yet-mature exclusions are all normal operation (green), reported via the run summary/log only. Then **security prioritization** (`advisories.ts`, once, after grouping): `fetchAdvisories` + `prioritizeGroups` stable-promote groups whose update resolves an OSV advisory to the front so the budget below spends slots on security fixes first — ordering only (identity untouched), fail-soft (an OSV outage keeps the neutral order), evaluated on the post-clamp `latest`. The resolved-advisory map also drives the PR body's Security column.
2. Count open depvisor PRs from the snapshot (`gh pr list --limit 1000`; the ceiling **fails open** if the snapshot is truncated or absent — locally, or on repos with more open PRs than the limit) → `newSlots = max(0, max_open_prs - openCount)`. For each group in stable order, first a **branch-collision guard** (distinct group keys can slugify to the same branch, e.g. `prod/@babel/core` vs `prod/babel-core`; colliders after the first are recorded `branch-collision`, red, and skipped so they cannot `ensureBranch`-reset the first group's commits away), then `classifyGroup` decides: skip-up-to-date / refresh / open-new / held-back (`held-back-by-limit`, green).
3. Between processed groups, reset the tree to base (`resetToBase` = `git checkout -f base` + `clean -fd`) and reinstall (`runInstall` with the resolved reset command) so node_modules returns to the base lockfile state. This reinstall also runs under `install_command: skip` (skip only skips the pre-agent install step); with `skip` **and** no committed lockfile no reset command exists, so every processable group after the first is recorded `reinstall-unavailable` (red — a fixable config gap, not the ceiling).
4. Baseline verification **per processed group** — `baseline-red` on the first processed group (base itself is broken), `reset-failed` on a later one (the reset was incomplete). Both are fail-closed, run-level stops.
5. Named agent session (`harness.session(\`group-<slug>\`)`) returning a **structured result** (Valibot `result` schema: summary / notable_changes / breaking_changes_addressed / residual_risks / verdict). notable_changes for packages outside the group are dropped at render time. The prompt's usage/model is projected into the group's GroupUsage here (before the verdict branch, so a defer reports its tokens too — display-only, see the status.ts bullet above).
6. Deterministic gates (`unexpected-commits`, `scope-violation`, verification) — authoritative regardless of what the agent claims.
7. Two commits → classify the committed `base..HEAD` diff (`test-changes.ts`, display only — a ⚠️ PR-body/step-summary warning when the agent touched tests, no gate) → classify per-package license changes from the fetched packuments (`license.ts`, display only — a ⚠️ PR-body warning when a package's `license` string changed, no gate) → emit a PR payload to `pr-preview/payloads/<NN>-<slug>.json` (a newly opened PR consumes a slot; a refresh does not). Status is written to `pr-preview/status.json` after every group.

- Per-group outcomes (`defer` → `deferred`, `no-structured-result`, `unexpected-commits`, `scope-violation`, `verification-failed`, `no-changes`, `branch-collision`, `reinstall-unavailable`, `release-age-unavailable`) record that group and **continue to the next group**; leftover changes from a defer are discarded deterministically, not trusted away. Only preflight, `bad-max-open-prs`, `bad-min-release-age`, `bad-min-release-age-exclude`, `bad-ignore`, `baseline-red`, and `reset-failed` stop the whole run.
- The composite action reports the status file after the token-holding open-pr step. Run-level `completed`/`no-updates` and group-level `pr-prepared`, `pr-up-to-date`, `deferred`, `open-pr-blocked` (a human took over the PR branch), and `held-back-by-limit` stay green; `baseline-red`, `reset-failed`, `bad-max-open-prs`, `bad-min-release-age`, `bad-min-release-age-exclude`, `bad-ignore`, `no-verify-scripts`, `scope-violation`, `verification-failed`, `missing-base`, `reinstall-unavailable`, `release-age-unavailable`, `branch-collision`, `open-pr-failed`, and the `in-progress` crash marker fail the job — a `completed` run is still red if any group failed — so silent no-PR outcomes notify users.
- `max_open_prs` is a ceiling on **open depvisor PRs** (Dependabot's open-pull-requests-limit model), so default `1` behaves ~like the old single-PR flow but stricter: it never opens a new PR while the ceiling is already met by pre-existing open PRs. Refreshing an existing PR (its targets drifted) is unlimited and consumes no slot.

### Invariants

Preserve these when changing the flow above.

- **Token separation is structural, not instruction-based**: the agent step holds only the LLM key; GitHub tokens exist only in the snapshot and open-pr steps, with `persist-credentials: false` on checkout. Never hand the agent step a token or let the agent run git/GitHub operations.
  - The multi-PR loop's group-boundary reinstall (`runInstall`) runs the target's lifecycle scripts inside the token-free agent step. This adds **no new exposure class**: the agent already runs `npm install <pkg>@<ver>` (with lifecycle scripts) during its own session, and no GitHub token is ever present in that step.
  - The user's checkout can defeat the separation from outside (actions/checkout defaults to `persist-credentials: true`), so `core/credentials.ts` fail-closes on persisted credentials in the checkout's repo-**local** git config: Authorization extraheaders (URL-scoped or not), userinfo in http(s) remote url/pushurl and in `url.*.insteadOf`/`pushInsteadOf` rewrites, `core.sshCommand`, repo-local credential helpers. The config read follows include directives (`--includes` in `localConfigEntries`) because checkout v6+ persists the token in a separate RUNNER_TEMP file referenced via `include.path` from the repo-local config. Enforced twice — the `check-credentials.ts` Action step, and the `persisted-credentials` preflight status covering local/non-action runs. Best-effort over known vectors, not a guarantee. Findings never include config values, and redact URL subsections from keys — both positions can carry the secret (`url.<https://token@…>.insteadOf` puts it in the key).
- **Branch name = stable group key = PR identity**: changing grouping or naming logic changes PR identity and breaks idempotency/skip-if-up-to-date.
  - The `minimum_release_age` clamp is a **known, documented exception**: a clamped candidate's `updateType` is recomputed, so a cooldown-window major can surface first as e.g. `prod/<name>` and later, once matured, move to `major/<name>` — a different branch and a new PR. The stranded earlier PR is not auto-closed and keeps occupying a `max_open_prs` slot until a human closes/merges it (README documents this; a superseded-PR closer is a recorded future feature, not present).
  - **Security prioritization (`advisories.ts`) is ordering ONLY, so it is NOT an identity exception**: it reorders groups but never touches keys, membership, or the installed version. Keep it that way — a "promote by editing the group key" shortcut would break idempotency. Its one determinism caveat is like the clamp's: OSV data is time-varying, so "same input → same order" holds only within an advisory snapshot; the sort itself is stable (`Array.prototype.sort`, Node 22) and deterministic given that snapshot.
- **Package-manager detection is pinned at preflight** (`core/pm.ts`, called once in `update.ts` preflight against the trusted base tree; also by `install-target.ts` and `dev/scan.ts`).
  - Supported: npm + pnpm + bun; yarn and ambiguous lockfile mixes fail closed (`unsupported-package-manager` / `ambiguous-package-manager`, no agent run). bun repos need the bun binary on the runner (`oven-sh/setup-bun`, version pinned — see README).
  - Never re-detect after the agent has run — lockfiles are agent-writable, so post-agent detection would let the agent switch which commands the trusted steps execute.
  - The detected `pm` drives collect, verify `run` prefixes, the agent's update-command instruction, the group-boundary reinstall in the multi-PR loop (via `pm.installCommand`, when `install_command` is `auto`/`skip`), and (via `pm.lockfiles`) the lockfile half of the two-commit split's bump commit — `git.ts`'s `manifestBumpPaths` pairs those lockfiles with every changed `package.json` (root or workspace), matched by basename.
  - `pm.updateInstruction(candidates, { pinExact })` is called with `pinExact: true` whenever the `minimum_release_age` cooldown is active: only bun's instruction changes (`bun add <name>@<latest>`, no `^`), because bun both writes the specifier verbatim and resolves ranges at install time — a caret range would let the install reach back into the cooldown window that the clamp just excluded. npm installs exact from `npm install <name>@<ver>` and pnpm from `pnpm -r update <name>@<ver>`, so they ignore the flag.
  - **Workspace monorepos (npm/pnpm/bun `workspaces`) are supported**: collect populates `Candidate.locations` (repo-relative workspace paths; root = `""`) and `pm.updateInstruction(candidates)` builds workspace-scoped commands from it (`npm install … -w <ws>`; a single `pnpm -r update …`; `bun add --cwd <ws> …`), so an update only ever touches the workspace(s) that already declare a dependency, never the root. bun needs a name→path bridge because `bun outdated -r` names the workspace but `bun add --cwd` wants the path — `collect.ts`'s `bunWorkspaceMap` expands the root `workspaces` globs (literal or single-level `dir/*`; deeper globs and unmapped workspace names fail closed). yarn workspaces stay out of scope.
  - The scope gate guards the corresponding surfaces, diffed against base: a path deny-list (PM config files, CI config, git hooks) plus guarded package.json fields. The exact lists are `DENY` and `GUARDED_FIELDS` in `core/scope.ts` — extend them when adding a PM or any new execution surface, and mirror path additions in `agents/updater.md` so the agent avoids them instead of burning a run on a scope violation.
    - **Tests are the one surface the scope gate cannot deny** (adapting a test to a changed API is a legitimate update), so they are handled by _visibility_ instead: `core/test-changes.ts` classifies the committed diff and surfaces a ⚠️ warning in the PR body/step summary. Not a gate — do not add a deny for tests.
- **The verification gate refusing to vouch (`no-verify-scripts`) means no PR**, by design.
  - Verify steps come from script auto-detection (`build`/`lint`/`test` — `typecheck` deliberately not detected), or from the `verify_commands` input (`DEPVISOR_VERIFY_COMMANDS`, newline-separated shell commands), which **replaces** auto-detection entirely.
  - `verify_commands` must only ever come from the workflow file/env, never from the agent-writable target tree.
  - The same gate also runs on the base tip **before the agent, once per processed group** (`baseline-red` on the first — base itself is broken; `reset-failed` on a later group — the between-groups tree reset was incomplete). Either is a fail-closed run-level stop, so post-update failures are always attributable to that group's update. The reset (`resetToBase` + reinstall) uses `git clean -fd` (no `-x`), so gitignored verify artifacts can survive between groups; per-group baseline re-verification is the safety net that catches a leak as `reset-failed`.
- **`.git/` is outside the scope gate (it only sees the working tree), so it is treated as attacker-writable.**
  - Every git invocation disables local hooks (`NO_HOOKS = -c core.hooksPath=/dev/null`, exported from `git.ts`, reused in `github.ts`). Never add a git call that bypasses `run()`/the `git()` wrapper.
  - The token-holding open-pr step runs entirely inside a fresh clone (`prepareCleanPush` in `github.ts`), because `.git/config` alone is a rich command-execution surface (`credential.helper=!cmd` fires on push auth, plus `core.fsmonitor`, `diff.external`, `filter.*`, …) that hook-disabling does not cover. Do not make the open-pr step operate on the target checkout's `.git`.
  - The push **target** is equally untrusted: in CI it comes from `DEPVISOR_REMOTE_URL` (Actions context `${server_url}/${repository}`); only when that is unset (local dev on a trusted machine) does it fall back to the checkout's `remote.origin.url` — the one read the open-pr step performs against the target checkout's `.git`. Either way it must pass `isNetworkRemote` — a push to a local/`file:`/helper remote would run that remote's server-side hooks in this `GH_TOKEN` process.
  - The payload's **`title`/`body` are untrusted at this boundary too** (written by the tokenless step): `openPrWithGh` re-sanitizes them at the exit (`sanitizePrBody`, preserving only a strictly-validated versions marker), just as it re-validates `base` and branch authorship.
  - The **inherited environment** is untrusted too (`local()` gives no isolation, so the agent can write `$HOME/.gitconfig` or inject via `$GITHUB_ENV`/`$GITHUB_PATH`): all git/gh in `github.ts` run via `buildSecureEnv` (token env allowlist only, clean HOME/XDG/GH config dir, `GIT_CONFIG_NOSYSTEM=1`, no inherited `GIT_*`/loader/exec env, PATH pinned, binaries by absolute path). Keep every subprocess in this module on that env.
- **Keep the Flue dependency thin**: `defineAgent` / `defineWorkflow` / `defineTool` / `harness.session(name)` (named per-group sessions) / `session.prompt` (incl. its `{ result }` structured-output option, and reading `.usage`/`.model` off the returned `PromptResultResponse` — no new symbol, so not a footprint expansion) / workflow `log` / `ResultUnavailableError`, exact-pinned beta.
  - `defineTool` + `{ result }` were a deliberate expansion for the release-notes tool and structured output; named `harness.session(name)` was the expansion for per-group session isolation in the multi-PR loop. Keep the rest thin so the AI-SDK escape hatch stays open.
  - All are "Flue lets you not write it" rather than "only Flue can express it," so portability is preserved.

### Development workflow

`.github/workflows/depvisor.yml` is the **development** workflow: it runs the composite action from the checkout via `uses: ./`, matching consumer workflows except for `uses: ./` and `install_command: skip` (the target is depvisor itself, a pnpm repo — the action's own `pnpm install` covers it).

The composite action has two documented GitHub-runner quirks (nested `uses:` doesn't evaluate `github.action_path`; pnpm/action-setup breaks absolute paths) — read the comments in action.yml before touching it.

## Gotchas

- Env config treats **empty strings as "not set"** everywhere (`||`/falsy checks, never `??` — see `update.ts`, `agents/updater.ts`, `shared/target.ts`, `open-pr.ts`): the composite action forwards unset inputs as empty strings. `DEPVISOR_MAX_OPEN_PRS` follows this too (empty → `1`, via `budget.ts`'s `parseMaxOpenPrs`; a non-positive-integer value fails the run with `bad-max-open-prs`), as does `DEPVISOR_MIN_RELEASE_AGE` (empty → `1`, `0` = explicit disable, via `release-age.ts`'s `parseMinReleaseAge`; anything else fails the run with `bad-min-release-age`), `DEPVISOR_MIN_RELEASE_AGE_EXCLUDE` (empty → no exemptions, via `release-age.ts`'s `parseMinReleaseAgeExclude`; an unrecognized entry fails the run with `bad-min-release-age-exclude`), and `DEPVISOR_IGNORE` (empty → no rules, via `ignore.ts`'s `parseIgnore`; an unrecognized entry fails the run with `bad-ignore`). `DEPVISOR_MODEL` has **no default** — the agent factory throws if it's unset/empty (action.yml fail-fasts earlier with a clearer `::error`, since GitHub doesn't enforce `required:` on action inputs).
- `npm outdated` / `pnpm outdated` read the **installed tree**: without an install in the fixture, candidates are hidden. Both exit 1 when updates exist (JSON still on stdout). The argv carry the flags that make workspace deps visible/locatable: npm uses `--json --long` (each entry gains `type` for dev/prod and `dependedByLocation` for the workspace path; a dep at differing versions across workspaces arrives as an **array**, merged into one candidate — lowest `current`, the distinct set retained in `currents` for advisory matching, union `locations`, dev only if every occurrence is dev), pnpm uses `-r --format json` (without `-r` only the root package's deps are reported; entries carry `dependencyType` and `dependentPackages[].location`, which is name-keyed so cross-workspace occurrences collapse to a single entry whose `current` is the **highest** installed version — the update is unaffected since `pnpm -r update` reaches every workspace, but the dropped lower versions are invisible to advisory matching, above). Hence the separate parsers in `collect.ts`.
  - `bun outdated` is the odd one out: it reads the **committed lockfile** (works without an install, errors without a lockfile), always exits 0, and has **no JSON output** — `parseBunOutdated` in `collect.ts` parses its ASCII table and fails closed on any format drift. depvisor always passes `-r`, so the table always carries a `Workspace` column (the declaring workspace's **name**, root included); the parser merges a package's per-workspace rows and resolves names to paths via `bunWorkspaceMap`. `collectCandidates` strips `FORCE_COLOR` from the spawn env: it flips bun to box-drawing + ANSI even when piped, and beats `NO_COLOR` (verified on bun 1.3.14).
- `fixtures/sample-app/` (npm), `fixtures/sample-app-pnpm/`, and `fixtures/sample-app-bun/` are gitignored throwaway git repos generated from the shared `sample-app.template/` (committed; it deliberately contains code that the lru-cache major update breaks). The pnpm variant gets its own `pnpm-workspace.yaml` written by `init-fixture.sh` — without it, pnpm walks up and captures the fixture into **depvisor's own workspace**, installing nothing locally. The bun variant needs no such guard (bun anchors workspaces on package.json `workspaces`, which depvisor lacks).
  - `fixtures/sample-app-workspaces/` (also gitignored) is the npm-workspaces monorepo variant, from its **own** committed template `sample-app-workspaces.template/` (two packages, `@sample/core` + `@sample/web`, sharing `semver` so collection merges it and the update scopes `-w` to both). The template ships no lockfile, so `--workspaces` generates the single root `package-lock.json` before the baseline commit (like pnpm/bun). Root `build`/`test` scripts fan out with `npm run … --workspaces --if-present`, which is what the verify gate detects.
  - `fixtures/sample-app-bun-workspaces/` is the bun-workspaces counterpart (`--pm bun --workspaces`), from `sample-app-bun-workspaces.template/`. Same two-package shape, but the root scripts fan out with `bun run --filter='*' …` and the baseline generates `bun.lock`. `@sample/web` declares its own `@types/semver` because bun's isolated linker (unlike npm's hoisting) won't surface `@sample/core`'s copy.
- No build step for the deterministic core: `src/core/`, `src/open-pr.ts`, `src/report-status.ts`, `src/install-target.ts`, `src/check-credentials.ts`, `src/dev/scan.ts`, and the tests run directly under Node with explicit `.ts` import extensions; tsconfig is maximally strict (`exactOptionalPropertyTypes`, `noUncheckedIndexedAccess`).
  - The **agent module is the exception**: `agents/updater.ts` uses a Flue-bundler-only `.md` import (`with { type: "markdown" }`), so it is loadable only under `flue run`/`flue build`, not plain `node` (`ERR_UNKNOWN_FILE_EXTENSION`).
  - Keep it that way — nothing on a plain-node path (tests, `open-pr.ts`, `report-status.ts`, `install-target.ts`, `check-credentials.ts`, `dev/scan.ts`, `core/`) may import an agent/workflow module.
- For Flue API questions, use the `flue` skill (`flue docs search/read`) rather than guessing — the dependency is a beta.
